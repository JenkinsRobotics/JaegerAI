# JROS 0.5.0 — the identity statement (+ the Mochi fold)

**Released:** 2026-06-20 (tag `0.5.0`; patch releases `0.5.1` / `0.5.2`
cut from the `0.6` branch afterward)
**Tag:** `0.5.0`
**Tagline:** JROS = Hermes-in-`agent/` + ROS-in-`nodes/` + a shared
`transport/` that lets them talk

0.4.0 made JROS a node framework. 0.5.0 is where the *agent* becomes
visibly alive on top of that framework — voice that behaves like the
keyboard, an animated face, a personality, a skill tree — and where the
separate **Mochi** project (the operator's character/animation engine)
is folded into one repo. After 0.5.0 there is one product, not two.

## What we set out to do

Three things:

1. Make the agent **transport-agnostic** — mic and keyboard should produce
   identical behaviour, with no voice-specific logic leaking into the brain.
2. Bring the **Mochi** work (characters, animation, the GUI, media + scene
   tooling) into JROS as first-class subsystems rather than a sibling repo.
3. Give the agent an **identity and a body it can express** — a structured
   personality, a character it plays, an animated avatar, and a skill tree it
   levels up.

## What actually shipped

### Voice — transport-agnostic agent
- **Removed the in-brain LLM voice gate** (`<reply>`/`<ignore>`). It made one
  model do two jobs (gatekeeping *and* tool-calling) and suppressed routing —
  `gemma-4-26B-A4B` routed 0/3 tool prompts with the gate on, 3/3 with it off.
  Ambient-speech filtering now lives in the voice INPUT layer (VAD + wake
  word), never the prompt. Keyboard and mic are now identical paths.

### STT method layer
- `plugins/whisper_stt/` reorganised into swappable **method subfolders**
  (`two_pass`, `continuous`, `local_agreement`) behind a single `STTAdapter`
  interface + a `registry.py` swap point (`config.stt_mode`). A one-command
  CLI bench (`python -m jaeger_os.plugins.whisper_stt`) reports load /
  transcribe / RTF / WER per method — the harness is the test.

### Animation + avatar
- L1–L4 animation adapters vendored from the operator's Mochi engine
  (`image` / `bitmap` / `sprite` / `gif` / `math`). `AnimationNode` +
  `FrameBridge` ship frames over a WebSocket to the in-tree Swift avatar app.
  End-to-end proven: agent → bus → adapter → bridge → client, byte-perfect.

### Timeline + skill tree + personality
- `TimelineRunner` — wall-clock multi-track scheduler (`/act/animation`,
  `/act/speech`, …). `SkillTreeRegistry` — XP-driven progression (prereqs,
  level-up, mastery cascade) wired to the bus by `XpEmitter`.
- `Personality` — structured persona (HEXACO + SPECIAL + Expression + Domains
  + speech patterns); `compose_block` produces the system-prompt fragment the
  brain reads each turn.

### Characters — the persona system (from Mochi)
- A **character** is the persona the agent *plays* (identity + lore + traits +
  assets + level/revision); `personality/characters/` ships 14 (GLaDOS, HAL,
  Jarvis, Mochi, Kamina, Simon, …). Identity / soul / name / voice all resolve
  from the **active character** — every agent always plays one (default
  `jarvis`), and a switch instant-applies (prompt rebuilt that turn).
- **Creation picks a character, not a prompt** — the wizard's identity-authoring
  step became a character picker; the persona-template path was retired.

### Mochi → Jaeger Studio
- Mochi's GUIs became **Jaeger Studio** (the PySide6 surface); the character
  system, animation/media nodes, the launcher, the test harness, and MScript
  scenes came across in the fold. One repo from here on.

### Patch releases off the 0.6 branch
- **`0.5.1`** — the agent truthfulness gate (research → confirm → ask → execute
  → test), `set_credential`, steer-on-every-message, in-process plugin bridges
  (Telegram/Discord/iMessage from the live agent), boot-time LLM prewarm, and a
  live activity trace in the windowed chat.
- **`0.5.2`** — autonomy modes (`ask` / `scoped` / `auto`) and admin-trust /
  channel-agnostic polish (carried forward into the 0.6 line).

## What we tried and abandoned

1. **The in-brain voice gate** — the headline removal above. A model judging
   "is this speech for me?" in the same prompt it routes tools regressed
   routing badly. Filtering belongs in the input layer.
2. **Per-instance authored personas** — replaced by reusable characters. An
   operator shouldn't hand-write `soul.md`; they should pick (or build) a
   character once and deploy it across agents.

## Lessons learned

- **One model, one job per prompt.** Every time we asked the brain to also
  gatekeep / classify in its system prompt, tool routing suffered. Keep the
  prompt about the task; push filtering to dedicated layers.
- **Fold sibling projects early.** Mochi and JROS were diverging; merging the
  character/animation/GUI work into one repo (rather than vendoring twice) is
  what made the "same brain, many bodies" story real instead of aspirational.
- **The harness is the test.** The STT bench and the animation pipeline were
  validated by their own runnable CLIs, not a separate suite — cheaper and
  closer to how they're actually used.

## Patterns absorbed

- **Mochi's character model** — persona-as-data (identity/lore/traits/assets)
  that a runtime *plays*, cleanly separated from the deployment that runs it.
- **VoiceLLM's input-layer filtering** — VAD + wake word gate *before* the
  brain, not inside it.

## Where 0.5.0 lives in the codebase today

- Voice: `jaeger_os/plugins/whisper_stt/`, `core/voice/`, `core/audio/`.
- Characters / personality: `jaeger_os/personality/` (+ `characters/`).
- Animation / avatar: `jaeger_os/nodes/`, `interfaces/avatar/`.
- Studio GUI: `jaeger_os/interfaces/studio/` (+ `interfaces/pyside6/`).
- Skill tree / timeline: `jaeger_os/core/` (skill-tree + timeline modules).

## What 0.5.0 sets up

A living agent with a face, a voice, a personality, and a place to grow
skills — and a single repo that holds the brain *and* the body tooling. That
foundation is what 0.6.0 wraps in a real product shell: a clean install, an
in-place update, autostart, and the operator lifecycle around the agent.
