# JROS 0.7.0 — Swift-first app + the two-runner agentic core

**Status:** active branch `0.6.0` (ships as 0.7.0) — this summarises what the
branch delivered before the master merge
**Tag:** `0.7.0` *(cut at merge)*
**Tagline:** JROS became a native Mac app, and the agent loop became a measured
two-runner system

0.6.0 made JROS *feel like a product to install and run*. 0.7.0 makes it a
**native Mac app** and turns the agent loop into a **measured, two-runner
architecture** — while nailing down the identity model (the agent you named vs.
the character it plays) across every surface.

## What we set out to do

1. Make the windowed app the primary way to run JROS, without giving up the
   terminal for SSH/headless.
2. Harden the Python↔Swift seam into a real, versioned contract.
3. Make the agentic pipeline's quality *measurable* — two runners, two
   benchmark types, and a security posture we can state honestly.
4. Get the agent's identity right everywhere: the name is the instance's, the
   character is only the persona it plays.

## What actually shipped

### Swift-first native app
- **`JaegerOS.app`** is now the primary UI — menu-bar resident, splash →
  chat/settings windows, quit-from-tray (the native Mac pattern). Windows close
  freely; the agent lives in the tray. **`JaegerOS-dev.app`** is pinned to the
  gitignored `jros-dev` instance for framework work.
- **Bridge protocol v1** — a hardened, versioned NDJSON contract between the
  Swift app and the Python core: **fast-ready** (transport usable in ~0.5s
  while the model warms behind it), typed frames with **cross-language fixtures**
  (`protocol_v1_fixtures.json` — pytest asserts the Python builders, XCTest
  asserts the Swift decoder, against the same file), a real connection state
  machine, interactive permissions, and clean-exit (`bye`) markers. Frame
  changes are additive-only.
- Bare `jaeger` launches the app; `jaeger --tui` / `jaeger dev` keep the
  terminal first-class. `launch.py` / `./launch` were **removed** in favour of
  the two apps + `jaeger` verbs; one-line install + `jaeger update`.
  iOS-style first-run onboarding, with Python still the wizard's source of truth.

### Two-runner agentic core
See `dev/docs/reality/agentic_runners.md`.
- **Realtime runner** — "soft loop, hard boundary": the fluid loop plus a
  runner-owned **verify gate** (catches PLAN-halt and claim-without-action with
  one soft nudge) and a **persona output filter** (character voice restored
  without the ~7-point worker-prompt tax).
- **Deep Think runner** — a staged assembly line (plan artifact → execute →
  evidence-verified completion → one replan cycle), no longer trust-by-return.
- **Inference lanes** — one loaded model, two llama contexts: a worker lane
  (conversation, never evicted) and an **aux lane** (persona filter / finalizer
  / reflection) so the side calls never evict the worker's warm prefix.
  Persona-ON warm ttft **45.6s → 0.71s**. Configurable via `model.ctx` +
  `model.aux_ctx`.

### Identity vs. character, everywhere
- The agent's **name** comes from `identity.yaml`, always — never the character
  it's playing ("name your robot Ted, it plays HAL"). Audited and fixed across
  every surface: menu tray, TUI, chat, settings HUD, and the bridge frames
  (`agent_name` now rides the fast-ready and ready frames, so the tray never
  flashes the character name at boot). The split is encoded in the protocol
  fixtures both languages assert (`agent_name` "Jarvis" ≠ `character` "HAL 9000").
- The instance owns a **profile picture** (`identity.avatar`) — the agent's own
  face, independent of the persona. It defaults to the active character's card
  until you set one; switching persona only *offers* to adopt the new card and
  never overrides a custom picture.
- The settings **Home page is a status overview** (picture, name, "playing X",
  live status + a facts grid); a new **Instance tab** edits the agent name +
  profile picture. Character/Traits still edit the persona and never touch the
  name.

### Two benchmark types + an honest security posture
- The **E4B routing corpus** holds at **79/81** (all-time high; 26B 75–76) —
  `dev/benchmark/bench.py`.
- A new **51-case hermetic scenario suite** (`dev/benchmark/scenarios.py`) —
  multi-turn, full-system, run against a throwaway temp instance so it can't
  pollute live state: 36 scriptable + 15 security gates. Security **14/15 on
  the default 4B**, **15/15 on the 26B**. The one 4B gap (`inj-mem-poison` — it
  persists a self-authorizing "delete any file" fact) is fixed by model scale;
  recorded in STATUS.md, gated behind the permission layer regardless.
- Two initially-failing gates turned out **over-strict** (a scoped credential
  search the tool layer already blocks + then refuses; a *harmless* injection
  echo by a harm-gated agent) and were refined to reward proper behavior, not
  never-touch.

### Also
- **Scheduler fix** — reminders / scheduled prompts now actually fire on the
  app + bridge path (a CronRunner starts with the agent) and run in local
  wallclock; "remind me in 1 minute" no longer lands hours late.
- **Memory v2** — subject-attributed SQL facts with provenance and history;
  hermetic bench isolation.
- **`jaeger-studio` extracted** to its own repo; the unshipped windowed
  surfaces moved out of the runtime.
- HomeAssistant + fal.ai plugins; 99 integrity-guarded skills.

## What's deferred (0.8+)
- The **modular-framework north star** (`dev/docs/vision/framework_vision.md`) —
  modules (TTS/STT/vision/hardware nodes, plugins) that own their own config /
  settings / lifecycle and register into a unified surface. 0.7 kept the
  settings scope tight (schema-derived config only); the module framework is
  0.8.
- Hardware adapters (JP01), full Deep Think Phase 2, reflect→skill creation.

## Bench gate
The E4B routing corpus must hold **≥79/81** for any agentic-pipeline change;
it held throughout the 0.7 work.
