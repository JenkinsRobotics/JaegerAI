# JROS 0.4.0 — node architecture

**Released:** 2026-06-06 (tagged + pushed; minor patches still
landing on `0.4.0` branch as of 2026-06-07)
**Tag:** `0.4.0`
**Tagline:** ROS + Agentic AI + Mac-first local hardware

This is the release where JROS stops being a single-process
agent and becomes a framework for embodied AI on a Mac.  The
brain stays one process; peripheral subsystems (TTS, STT, vision,
motors, lights) become bus-addressable nodes behind clean adapter
Protocols.

## What we set out to do

Two things, both motivated by 0.3.0's lessons:

1. **Get JROS ready for real hardware.**  JP01_Firmware exists; it's
   a real robot the operator built; JROS should eventually run on
   the Mac talking to JP01's Jetson + Teensy + ESP32 boards.  That
   means a transport layer that handles single-process today and
   multi-process / multi-board tomorrow without changing the
   agent loop.

2. **Stop hand-rolling the voice pipeline.**  0.3.0 absorbed VoiceLLM's
   `<ignore>`/`<reply>` gate; the operator wanted JROS to feel as
   responsive as VoiceLLM's reference loop.  That means matching
   not just one pattern but the whole conversational shape —
   pending queue, follow-up retry, self-speech filter, all the
   M3 orchestrator patterns.

The position statement that locked in early:

> **JROS = ROS + Agentic AI + Mac-first local hardware.**

No one else owns this combination — ROS doesn't think about LLM
agents; Hermes / OpenAI Agents / LangChain don't think about
embodiment; LM Studio / Ollama stop at chat.  JROS is the
intersection.

## What actually shipped

### Track A — node infrastructure

* **`topics.py` SSOT** — 11 typed topics (`/sense/audio_in`,
  `/sense/transcript`, `/sense/vision`, `/sense/touch`,
  `/sense/proprio`, `/sense/spoken`, `/act/speech`,
  `/act/audio_out`, `/act/motion`, `/act/light`,
  `/act/speech_stop`) as `msgspec.Struct` schemas with a common
  envelope (topic, topic_v, t_emit_ns, seq, node_id,
  correlation_id).

* **codec.py** — JSON for text topics, MessagePack for binary
  topics.  Picks per topic so the Bus implementations don't have
  to know.

* **InProcBus + ZMQBus + XPUB↔XSUB broker** — abstract Bus
  interface, two implementations, plus the broker that bridges
  separate ZMQ contexts (the foundation for `--mode multiprocess`
  whenever launch.py grows full subprocess orchestration).

* **Node base class** — setup → tick → teardown → health
  lifecycle, signal handling, log routing, exception isolation.

### Track B — node implementations

* **TTS node** (`/act/speech` → Kokoro → `/sense/spoken`) with
  barge-in via `/act/speech_stop`.

* **STT node** (Whisper → `/sense/transcript`).  Wraps
  `WhisperSTTContinuous` so the existing 0.3 voice features
  (wake word, follow-up window, mic-pause, AEC) carry over.

* **Vision node** (`/sense/vision` raw camera frames; no
  inference).  Two universal backends: `USBCameraAdapter`
  (cv2.VideoCapture) + `TCPCameraAdapter` (generic 4-byte-
  length-prefix protocol).  Operator was specific that vision
  = camera stream, not analysis; future inference is a separate
  downstream consumer of `/sense/vision`.

* **Motor + Light node skeletons** (Track C): universal
  Protocols + reference `SerialMotorAdapter` /
  `SerialLightAdapter` with ASCII line wire formats.  Per-board
  specifics (JP01-MC01 ESP32, JP01-AVC01 Teensy) land at
  instance level when JP01 actually wires up.

### Brain integration

* **Agent's `text_to_speech` tool routes through the bus.**  Tool
  signature unchanged (`text_to_speech("hi")`); under the hood
  it now publishes `SpeechCommand` and waits for `SpokenAck`.
  Pattern: *"a tool does the networking, a node does the
  execution"* (operator-locked contract).

* **Voice loop fully migrated** — STT phrases on
  `/sense/transcript`; TTS via `bus.request(SpeechCommand)`;
  barge-in via `/act/speech_stop` published from the STT
  callback.

* **TUI voice path** — was a release-blocker bypass (caught by
  the independent code review); migrated in the 0.4.0-followup
  patch round so `/voice on` in the TUI now exercises the bus
  end-to-end.

### Voice defaults aligned with VoiceLLM

Default flips, validated by side-by-side testing against the
reference voice_assistant_persistent.py:

* `wake_word: True` → `False` — LLM gate handles
  addressed-to-me far more robustly than wake-phrase
  transcription matching (base.en accuracy ceiling is the
  problem).
* `llm_gate: False` → `True` — every reply prefixed
  `<ignore>` / `<reply>`; `<ignore>` suppresses TTS.
* `barge_in: True` → `False` — mic-pause during TTS matches
  the reference's self-speech rejection strategy.
* `follow_up_seconds: 15.0` → `10.0` — shorter window = less
  time for stale noise between turns.
* STT thresholds aligned (`silence_hangover_ms: 1000 → 700`,
  `max_speech_ms: 12000 → 8000`).
* Audio backend default flipped from `avaudio` to `sounddevice`
  because the reference uses sounddevice and validated cleanly;
  avaudio backend has a known mic-capture divergence (deferred
  to 0.4.x).

### VoiceLLM orchestration patterns absorbed

* **Pending-phrase queue** — phrases arriving during a turn
  survive to the next iteration (opt-in via
  `voice.pending_queue: true`, default OFF).
* **Active-follow-up retry** — `<ignore>` during the follow-up
  window triggers one retry (default ON).
* **Self-speech similarity filter** — drops transcripts too
  similar to the agent's last reply (default OFF — mic-pause
  + LLM gate cover this in normal operation).

### Other

* **MLX-LM backend toggle** — schema now allows
  `backend: mlx_lm`; dispatcher already existed in `main.py`.
  Operator can install `mlx-lm` and point at an MLX model
  directory for 1.5–2× faster decode on Apple Silicon.

* **Model resolver checks LM Studio cache** — if the GGUF is
  already in `~/.lmstudio/models/...`, JROS uses it instead of
  re-downloading (operator hit this exactly with the 12B
  download → 7 GB redundant traffic).

* **Test surface: 1824+ tests** (was 1675 at 0.3.0).  New
  packages: `dev/tests/jaeger_os/transport/` (codec / InProcBus
  / ZMQBus / broker) and `dev/tests/jaeger_os/nodes/` (base /
  runtime / tts / stt / vision / motor / light).

## What we tried and abandoned

### 1. Pydantic everywhere

Initial plan had `topics.py` using `pydantic.BaseModel`.  Operator
flagged the inconsistency: VoiceLLM's Bus uses `@dataclass` for
speed, JROS was about to pay a 10× hot-path overhead for
consistency with the rest of the codebase.

**What replaced it:** `msgspec.Struct` for transport schemas,
Pydantic kept for config / manifests / tool schemas.  The
boundary is principled (hot-path = msgspec, validation-rich =
Pydantic) and the independent reviewer signed off on it.

### 2. STT/TTS in the brain

Initial node sketch had STT and TTS as in-process plugins the
brain called directly.  Operator pushed back: *"audio pipelines
will evolve; today's Kokoro becomes tomorrow's MLX-TTS without
touching the brain."*

**What replaced it:** STT and TTS are their own nodes from the
start.  The agent's tool surface is unchanged; the
implementation routes through the bus.  This made the
correlation-id contract (request → ack with matching cid) the
canonical pattern for tool-routed work.

### 3. `--mode multiprocess` end-to-end at 0.4.0

The XPUB↔XSUB broker is in (6 tests pass; cross-Bus bridge
proven).  The end-to-end operator workflow (`./launch
--mode multiprocess` spawns each node as a subprocess) deferred
to 0.4.x.  Wiring subprocess orchestration into `launch.py` is
its own piece of work that's separable from the architectural
decision.

### 4. WebRTC VAD option

Noted in VoiceLLM library review as a future-quality candidate
(more robust than energy-gating).  Adding it as a `vad_mode:
energy | webrtc` toggle requires a `WhisperSTTContinuous`
refactor (~150-200 lines).  Deferred to 0.4.x; the operator
agreed.

### 5. Explicit orchestrator FSM

Voice loop's IDLE → THINKING → RESPONDING → IDLE state is
implicit (scattered booleans).  VoiceLLM has it as a real state
machine; would aid testability + debuggability.  Deferred as a
refactor.

## Lessons learned

1. **Independent code review found things the implementing
   agent missed.**  The reviewer caught:
   - TUI voice was bypassing the TTS node entirely (the
     headline release claim was false until the patch round
     fixed it)
   - voice_loop ignored fresh-install defaults (CLI overrode
     config instead of the other way around)
   - InProcBus.publish() blocked despite the contract saying
     non-blocking
   - ensure_tts_node was under-tested
   - SpeechStop was global, no correlation_id matching
   - Vision topic naming was misleading (should be
     `/sense/camera_frame`, `/sense/vision` reserved for
     inference)

   Every one of these was real; every one made it into the
   patch round.  Two reviews — the planning review at
   2026-06-04 (which caught the msgspec inconsistency + the
   codec.py-as-its-own-step idea) and the post-release review
   at 2026-06-06 — both materially improved the release.

2. **"No broken step" cadence worked.**  Each step in Track A
   landed with its own commit + tests + green suite.  When the
   reviewer's findings came back, the fix-it round was
   contained per-finding instead of mega-commits.

3. **Defaults matter.**  0.3.0 shipped LLM gate as opt-in;
   0.4.0 needed operator-validated defaults flipped because the
   operator's actual VoiceLLM test produced 1000× better feel
   with `wake_word: False` + `llm_gate: True`.  Don't let the
   operator-friendly defaults trail the proven-good
   configuration by a release.

4. **Universal interfaces + per-instance specifics.**  Track C
   skeletons (Motor + Light Protocols with ASCII serial
   reference adapters) shipped because the operator was clear
   that JROS-the-library stays vague and JP01-specific wire
   formats live at instance level when integration happens.
   This is a real architectural commitment, not a punt — the
   universal interface IS the value.

5. **Local Gemma 12B is a real model.**  After the Metal OOM
   forced a switch from 26B to 12B, voice mode worked
   comfortably with 16 GB GPU headroom.  12B was leaderboard #1
   at the 0.3.0 routing bench; on M-class hardware it's the
   right "voice agent" model.  26B is overkill for voice
   conversation flow.

6. **The test isolation bug was a hidden cost of caching.**
   The model_resolver patch + a successful download dropped a
   real 7 GB GGUF into `<repo>/.jaeger_os/models/`, which
   `test_local_discovery.py` then picked up.  9 tests started
   failing because of a fixture that monkeypatched
   `_in_tree_models_path` but missed `_operator_state_models_path`.
   Lesson: test fixtures need to isolate ALL the scan paths,
   not just the ones the original test author knew about.

## Patterns absorbed

From [VoiceLLM](../library_review/voicellm.md):

* The 30-line `queue.Queue` Bus — became `InProcBus`.
* `<ignore>` / `<reply>` LLM-gated speech.
* Pending-phrase queue with last-write-wins.
* Active-follow-up retry.
* Self-speech similarity filter via `SequenceMatcher.ratio()`.

From [JP01_Firmware](../library_review/jp01_firmware.md):

* Topic-namespace convention (`/sense/*` for inputs, `/act/*`
  for commands).
* Board-naming corrected from the operator's clarification —
  JP01-AVC01 (Teensy) = audio + LED, JP01-MC01 (ESP32) = motors.

From the reviewing agent (across two reviews):

* `msgspec` for transport / Pydantic for config — the
  principled boundary.
* `codec.py` as its own step (the implementing agent's first
  Track A plan went topics.py → bus directly; the reviewer
  suggested codec.py as the layer between them).
* The full "tool does the networking, node does the execution"
  contract — naming the pattern made it enforceable.

From [Hermes](../library_review/final_review.md):

* The `supervisor.py` pattern — preserved at
  `library_review/hermes_supervisor.py` for Track D when it
  lands.

## Where 0.4.0 lives in the codebase today

This is the **current release**.  Everything described above
is in tree on the `0.4.0` branch and pushed to
[origin/0.4.0](https://github.com/JenkinsRobotics/JROS/tree/0.4.0)
and tagged as `0.4.0` + `v0.4.0`.

The brain process boots through `./launch` exactly as in 0.3.0
from the operator's point of view.  Underneath, the TTS node,
STT node, runtime singleton, and bus are all live — visible in
the boot log as `[node:tts] subscribed to ...` lines.

`master` on the remote is still at the 0.3.0 release commit;
the 0.4.0 force-push to master is gated on explicit operator OK
per the standing rule.

## What 0.4.0 sets up

* **Track D** (supervisor, health bench, backpressure tuning)
  — preserved hermes_supervisor.py is ready to port onto the
  Node base class.
* **JP01 hardware integration** — Track C skeletons are the
  universal interface; per-board adapters land when the
  operator wires JP01 boards in.
* **`--mode multiprocess` end-to-end** — broker shipped; the
  launch.py orchestration is the remaining piece.
* **MLX-LM as a real performance option** — schema allows it;
  operator can flip when they want to test.
* **A second pass at the VoiceLLM-comparable voice UX** — the
  patterns absorbed at 0.4.0 should make the gap small;
  WebRTC VAD + explicit FSM would close it further.
