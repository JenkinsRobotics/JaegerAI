# JROS 0.3.0 — the refactor release

**Released:** 2026-06-06
**Tag:** `0.3.0` (commit `17b9d90`)
**Tagline:** voice pipeline rebuild + skill v3 + persona prefill

This is the release with the biggest lessons.  Two large initiatives
got fully built and then walked back from before they shipped.  The
walked-back code is still in tree as archived — preserved
deliberately so the lessons are visible rather than buried.

## What we set out to do

After 0.2.6 added the messaging gateway + daemon mode, the next
release was originally planned (in
[`odysseus_review_and_0.3.0_plan.md`](../odysseus_review_and_0.3.0_plan.md))
to be **"JROS becomes a desktop product"**:

* A Swift / SwiftUI native macOS desktop app (`apps/JaegerOS/`)
* A daemon-attached architecture where every client (CLI, tray,
  voice loop, messaging bridges, Swift GUI) talks to one running
  Gemma via socket RPC
* A rich_tui terminal client that also attached to the daemon
* The tray icon as a real menu-bar surface launching all of the above

By the time the operator was deep in this, the actual pain points
were:

1. **The 0.2.6 audio pipeline was wedging CoreAudio.**  Long-running
   voice sessions on macOS 26.5 would slowly accumulate PortAudio
   internal errors (`-9986`) and eventually segfault at
   interpreter exit (`Pa_Terminate` in `__del__`).
2. **The Swift desktop app added work that didn't materially
   improve the operator's actual day-to-day experience.**  The TUI
   was already the operator's surface; the Swift app was solving
   a problem the operator didn't have.
3. **Daemon-attached multi-client added complexity** (socket layer,
   client SDKs, ipc semantics) for what amounted to "save one
   Gemma instance in RAM" — fine in theory, source of more bugs
   than it fixed in practice.

So 0.3.0 became a **refactor release** instead of a feature release.
What got fully built first then walked back lives in
[`0.3.0-archive` branch](https://github.com/JenkinsRobotics/JROS/tree/0.3.0-archive).
What actually shipped is documented below.

## What actually shipped

* **Persistent Kokoro player** (`plugins/kokoro_tts/persistent_player.py`)
  — one long-lived output stream per session, replaces the
  per-call `sd.OutputStream` open that was producing PortAudio
  errors.  Mirrors the proven
  `dev/tools/audio_smoke/voice_assistant_persistent.py`
  reference.

* **Dual audio backend** — sounddevice (PortAudio, default) and
  avaudio (PyObjC AVAudioEngine, Apple-native).  Both are persistent;
  operator can toggle via `config.voice.audio_backend` or
  `JAEGER_AUDIO_BACKEND` env var.

* **AVAudioEngine bridge** (`plugins/avaudio_io/`) — Python ↔ PyObjC
  wrappers for Apple-native input + output streams.  Bypasses the
  whole PortAudio segfault class on macOS.

* **Whisper STT hardening** — energy gating, AEC plumbing, non-
  speech-marker filter (`[BLANK_AUDIO]`, `(beep)`, etc.) so the
  agent doesn't try to respond to noise.

* **Skill system v3** — unified `manifest.json` schema with
  capability scoring bands.  See `jaeger_os/skills/v3/`.  Skills
  declare what they're good at (routing / multistep / cross-turn /
  safety) and the agent picks based on the active toolset.

* **Persona prefill framework** — `personas/<name>/` templates the
  wizard can offer at setup.  Each carries `identity.yaml` +
  `soul.md` + a starter skill bundle.

* **Gemma 4 12B added to the model registry** — `gemma-4-12b-it-q4_k_m`
  is the 24 GB tier's deep-think pick.  Bench-validated at 94.9%
  routing (leaderboard #1 at the time).

* **Bench infrastructure fix** — writer + aggregator dir-mismatch
  fix (Codex's FIX 4); model identities normalised to lowercase
  with hyphens.  Old dirs preserved at
  `dev/benchmark/flat/gemma-4-26B-A4B-it-Q4_K_M/`.

* **`./launch` sandbox launcher** — single command boots the
  in-repo sandbox with a real-verification boot scroll
  (model check, AVAudio bridge import, Whisper assets, Kokoro
  package, skill registry, TUI module).  Replaces the prior
  cosmetic boot output.

* **LLM-gated speech** (`<ignore>`/`<reply>`) absorbed from
  VoiceLLM.  Opt-in at 0.3.0 via `config.voice.llm_gate: true`;
  the system prompt picks up `VOICE_LLM_GATE_RULE` when the
  env-var indicator is set.

## What we tried and abandoned

The big ones:

### 1. Swift / SwiftUI desktop app — `apps/JaegerOS/`

The full plan was in `apps/JaegerOS/` (now archived in
`0.3.0-archive`): menu-bar status item, chat window, floating
"pill" mode, voice capture via AVAudioEngine, AppleSpeech STT
fallback, AVSpeechSynthesizer TTS fallback.

**Why we walked back:** the operator's day-to-day surface is
the TUI.  The Swift app was solving a "JROS needs a real native
surface to feel like a product" problem that, on closer look,
wasn't actually present.  The TUI is what got tested; the Swift
app was where the bugs were accumulating.

**What we kept from it:** the `voice_assistant_avaudio.py`
reference (`dev/tools/audio_smoke/`) — that was the design
study that produced the `avaudio_io` PyObjC bridge.  The Swift
work informed the bridge but the Swift surface itself isn't
needed.

### 2. Daemon-attached rich_tui

Built fully: a daemon-attached rich TUI client that opened a
ZMQ socket to the running Gemma daemon, streamed turns in real
time, displayed tool activity in a side panel.

**Why we walked back:** for one operator on one machine, the
single-process TUI is strictly simpler.  The daemon-attached
shape only earns its keep when you have multiple clients
(Swift GUI + voice loop + messaging bridges).  Dropping the
Swift app collapsed the case for the daemon split.

**What we kept:** `jaeger_os/daemon/` stays in tree as archived
code.  Not wired into `./launch`; not removed.  If we ever
need cross-client model sharing it's there.

### 3. PortAudio for long-running TTS

The 0.2.6 path opened a new `sd.OutputStream` per `speak()`
call.  Under macOS 26.5 this leaked CoreAudio device handles,
eventually wedged the audio subsystem, and segfaulted at
`Pa_Terminate` on interpreter exit.

**What replaced it:** the persistent Kokoro player.  One stream
per session.  Chunks streamed into it as Kokoro produces them.
The `_close_stream()` only runs at shutdown.

### 4. Cosmetic boot scroll

The prior launch printed colourful boot messages without
actually checking the things it claimed to check.  Operator
caught this — a boot scroll that LIES is worse than no boot
scroll.  Replaced with `./launch`'s real-verification version
where every line is a check that actually runs.

## Lessons learned

1. **Walk the user flow before shipping.**  The Swift app + daemon
   work would have shipped on inspection alone.  The walk-through
   (does the operator actually USE this surface?) is what caught
   it.  Now memorialised in the project memory as
   "Walk user flows before shipping" — see
   `~/.claude/projects/.../memory/feedback-walk-flow-before-shipping.md`.

2. **0.2.0 → 0.2.6 was rapid iteration; 0.3.0 should have been
   the same.**  Instead the Swift + daemon work bunched up and
   produced a release that was 80% walked-back code.  Smaller
   release cuts catch the wrong-direction work earlier.

3. **Persistent streams + deterministic teardown matter on
   macOS.**  PortAudio on macOS doesn't tolerate per-call open/
   close at scale.  Either keep streams alive for the whole
   process OR use AVAudioEngine (which is much more tolerant).

4. **The bench was a leading indicator the architecture was right.**
   55/59 routing pass rate before and after the rebuild — no
   regression at the agent level even with the audio pipeline
   entirely swapped.  When the agent loop is clean, swap the
   audio layer all you want.

5. **Codex's audit caught what fresh eyes catch.**  7 fixes (FIX 1
   drain timeout HIGH, FIX 2 docstring, FIX 3 barge-in HIGH, FIX
   4 bench naming, FIX 5 launch --health, FIX 6 .gitignore, FIX 7
   installer) — the implementing agent missed several real things
   the reviewer caught.  This is the precedent that made the
   0.4.0 review prompt feel natural.

## Patterns absorbed

* The persistent Kokoro player pattern came from the proven
  `voice_assistant_persistent.py` reference (which itself
  predated JROS — it was an isolated voice loop the operator
  built earlier).
* The `<ignore>` / `<reply>` LLM gate came from VoiceLLM
  ([`library_review/voicellm.md`](../library_review/voicellm.md))
  — first time JROS deliberately absorbed a pattern from a
  sibling project.
* AVAudioEngine usage patterns came from reading Apple's
  AVFoundation docs + watching how Zoom / FaceTime handle the
  same constraints.

## Where 0.3.0 lives in the codebase today

The shipped 0.3.0 work is **almost all still current**:

* Persistent Kokoro player + dual backend — yes.
* AVAudioEngine bridge — yes.
* Skill system v3 — yes.
* Persona prefill — yes.
* Gemma 12B in the registry — yes.
* `./launch` — yes (extended with 0.4 flags like `--node-test`,
  `--tts-test`).
* LLM gate — yes (default ON in 0.4.x; was opt-in at 0.3.0).

The walked-back work is at
[`0.3.0-archive`](https://github.com/JenkinsRobotics/JROS/tree/0.3.0-archive)
on the remote; in the working tree, `apps/JaegerOS/` and
`jaeger_os/daemon/` exist as archived code (not wired into
boot, not removed).

The 9 walked-back upstream commits (from when origin/master had
diverged with Swift + daemon + tray work) live on
`origin/0.2.6` and in master's reflog — see commit `ca60f22`
for the rationale.

## What 0.3.0 set up for 0.4.0

The lessons from 0.3.0 directly shaped 0.4.0:

* "Walk the flow first" → 0.4.0's no-broken-step cadence (write
  → test → commit → wait; don't bundle).
* "Persistent + deterministic" → 0.4.0's Node base class with
  setup/tick/teardown lifecycle.
* "VoiceLLM patterns are worth absorbing" → 0.4.0 absorbed the
  Bus pattern, the LLM gate, the pending queue, the follow-up
  retry, the self-speech filter.
* "Daemon-attached multi-client was too much for one operator"
  → 0.4.0's single brain process with bus-addressable peripheral
  nodes (not daemon-attached anything).
