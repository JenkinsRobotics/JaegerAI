# JROS 0.2.0 → 0.2.6 — the 0.2.x line

**Released:** 2026-05-31 (all six minor versions cut on the same
day in rapid succession after the architecture work)
**Tags:** `0.2.0` through `0.2.6`
**Tagline:** sleep-cycle architecture + tier-aware wizard + bench v1.1

The 0.2.x line was a long single architecture push capped by a
series of small distribution polish releases.  Six tags in one day
sounds frantic; in practice the architecture work happened over
the preceding week and 0.2.1 → 0.2.6 were each one focused
fix-and-cut after operator feedback.

## What we set out to do

After 0.1.0 proved the agentic-local thesis worked, the next
question was *"how does JROS scale beyond one operator's machine,
and how do you describe what JROS even is?"*

Three concrete goals:

1. **Separate framework code from operator state.**  A clear
   answer to "what gets backed up vs. what gets `git pull`-ed".
2. **Sleep-cycle: the agent has an idle mode that does maintenance
   work.**  Deep think on a slower asleep model; promote skills;
   regenerate prompts; tidy memory.
3. **Make the bench corpus stable.**  0.1.0's 51-case corpus had
   drifted; pin a versioned 59-case corpus and run regressions
   against it.

## What actually shipped

**0.2.0 (core architectural release):**

* The **sleep-cycle architecture** — a separate model role
  (`asleep`) that wakes during idle time for deep think.
  ([`deep_think_design.md`](../core/deep_think_design.md))
* The **tier-aware setup wizard** — picks asleep model + skills
  based on the operator's hardware tier (8/16/24/32+ GB).
* **Bench v1.1**: 59 cases, versioned corpus, leaderboard format.
  See `dev/benchmark/HISTORY.md`.
* The **System / Runtime / User architecture** —
  `jaeger_os/core/` for framework, `~/.jaeger_os/instances/<name>/`
  for operator state.  Documented in
  [`architecture/system_runtime_user.md`](../reality/system_runtime_user.md).

**0.2.1 (architecture formalisation):**

* The architecture document became the official spec.  The
  three-bucket split (System / Runtime / User) became the load-
  bearing mental model the rest of JROS organises around.

**0.2.2 (model resolver + manifest follow-ups):**

* Pull the model-path resolution out of every framework's
  `main.py` into a single
  [`model_resolver.py`](../../jaeger_os/core/models/model_resolver.py).
* The model registry (GGUF lookups, hardware tier → asleep
  model recommendations).

**0.2.3 (distribution):**

* The **git-clone install model** — JROS isn't a `pip install`
  package; it's a repo you clone.  Operator state stays in the
  clone's own `.jaeger_os/` so upgrades are `git pull`.
* The **one-line curl installer** — a shell script that clones,
  builds the venv, and runs the setup wizard.

**0.2.4 (operator-facing polish):**

* Agent management subcommands (`./run.sh setup <name>`,
  `./run.sh list`, etc.).
* Curl installer Python detection fix.

**0.2.5 (wizard hardening):**

* Wizard EOFError fix when stdin had no terminal (CI / pipe).
* Dropped a silent force-backup in the setup subcommand that
  was masking existing-instance collisions.

**0.2.6 (the unification release):**

* **Messaging gateway with `--attach` flag** — Discord /
  Telegram / iMessage bridges that share the running daemon's
  loaded model instead of loading their own.
* Voice loop wake-word UX: "ok jaeger" / "hey jaeger" + phonetic
  variants required by default (Google-Home-style).
* AEC barge-in via speexdsp when available; mic-pause when not.
* Audio backend split: avaudio (Apple-native) on macOS,
  portaudio (sounddevice) elsewhere.
* The launch + tray work that would later get walked back at 0.3.

## What we tried and abandoned

* **Tracking agent activity in JSON files at scale.**  Worked at
  0.1.0; started breaking under realistic memory loads.  SQLite
  migration moved to a follow-up (landed in the 0.2.x line as DB
  Group 1 → 7).
* **A "thinking" tool for the agent.**  The agent would dispatch
  a `think` tool that did nothing operationally but signalled to
  the operator that work was happening.  Removed — confused more
  than it helped; replaced by real status messages.
* **The 0.1.0 51-case bench corpus.**  Pinned for a couple of
  weeks during the architecture work; superseded by the
  formal 59-case 1.1 corpus.  0.1.0 runs sit at the bottom of
  HISTORY.md for historical comparison.

## Lessons learned

1. **The three-bucket split was load-bearing.**  Saying "the
   framework is THIS dir, the operator's state is THAT dir, and
   they touch through documented seams" made every subsequent
   refactor easier.  Before this, every change risked
   accidentally moving operator state into framework code.
2. **Sleep-cycle deep think only works if the asleep model is
   different from the awake model.**  Using the same model for
   both meant the asleep work either blocked the awake side or
   confused the conversation state.
3. **The curl installer is psychologically important.**  A
   `git clone && cd && ./install` flow is what makes operators
   actually try the framework.  Anything heavier becomes a
   threshold.
4. **Wake-word matching at base.en accuracy is fragile.**  We
   shipped it anyway in 0.2.6 because the operator wanted
   Google-Home UX.  The fragility wouldn't bite until 0.4 when
   the operator tested side-by-side against VoiceLLM and saw
   how much better LLM-gated continuous-listen was.
5. **Releasing six tags in a day is fine if each is a clean
   single fix.**  Each had a focused commit set, a clean diff,
   and a real reason to be its own version.  The CHANGELOG
   preserves the why-per-tag.

## Patterns absorbed

* The instance-state directory layout came from looking at how
  Docker isolates per-container state.
* The wizard's tier detection came from reading LM Studio's
  hardware-aware model picker.
* The bench format (versioned corpus + leaderboard + chronological
  log) came from looking at lm-evaluation-harness.

## Where 0.2.x lives in the codebase today

Most of 0.2.x is **load-bearing current code**:

* The three-bucket architecture is exactly how the codebase is
  still organised — System (`jaeger_os/`), Runtime
  (`jaeger_os/main.py` + `core/`), User
  (`~/.jaeger_os/instances/<name>/`).
* The sleep-cycle deep-think model role is still how 0.4 picks
  the coder/deep-think model.
* The bench v1.1 corpus is the current bench.  Every model run
  since 2026-05-29 is in `dev/benchmark/HISTORY.md`.
* The git-clone install + curl installer is still the canonical
  install path.

What changed at 0.3+: the daemon-attached architecture +
messaging-gateway `--attach` flag work was walked back from in
0.3.0 (see [`0.3.0.md`](0.3.0.md)) — the daemon stays in tree as
archived but isn't the default operator flow.
