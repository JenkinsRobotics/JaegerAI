# 0.5.0 Swift renderer plan — `apps/JROS-Avatar/`

**Status:** scaffolding; iterative implementation throughout 0.5
**Operator-locked 2026-06-08:** Swift app for Mac-native rendering;
ships with the framework at `apps/JROS-Avatar/`.

## What this is

A standalone Mac app that connects to the running JROS Python brain
via WebSocket and renders the avatar's animation frames.  Out of
process from the brain — clean separation, can run in another
window, can be captured by OBS, can become the literal robot face
on a Jetson-attached display when 0.6 lands.

## Why Swift (and not Qt / browser)

* **Mac-native** — operator's daily driver is macOS; SwiftUI +
  AppKit are first-party.  No Qt baggage, no browser engine.
* **Performance** — Core Animation + Metal where needed.  Direct
  pixel buffer rendering with no JavaScript boundary.
* **Aesthetics** — the operator wants a "video-game aesthetic"
  visualisation eventually.  Swift is the right tool for native
  macOS visual polish.
* **Future hardware avatar** — same Swift app + Catalyst/iPad path
  could later run on an iPad attached to a JP01 robot face.
  Optional but the door is open.

## Wire protocol — frame messages

Each frame from Python is a single WebSocket binary message:

```
[ 4-byte big-endian header length L ]
[ L bytes of JSON header (UTF-8)    ]   { "w": 480, "h": 320,
                                          "format": "RGBA8",
                                          "duration_ms": 33,
                                          "is_final": false,
                                          "asset": "faces/smile.json" }
[ w * h * 4 bytes of RGBA pixel data ]
```

* JSON header is small (<200 bytes typical) — handler params, asset
  path for debugging, optional skill_id for XP visualisation.
* Pixel buffer is the actual frame.  No per-frame allocation in
  steady state — the Swift side reuses one `CGContext` of the
  correct size.

State events ride a separate text channel:

```
ws://127.0.0.1:8765/state    ←  AnimationState, TimelineProgress as JSON
```

## Phased delivery

### Phase 1 — bring the window up (this commit's scaffold)

* SwiftPM package
* Single-window SwiftUI app with a placeholder Canvas
* Connect button + status indicator
* WebSocket plumbing stubbed (logs received messages)
* No real rendering yet

### Phase 2 — render frames

* `FrameDecoder` turns `(header, bytes)` → `CGImage`
* `RendererView` displays the latest frame
* Frame rate reported on status bar

### Phase 3 — Mochi adapters integrated end-to-end

* JROS publishes a static-image command; Swift renders
* JROS publishes a gif command; Swift renders animation
* JROS publishes an mp4 command; Swift renders video
* Lip-sync wiring: `/sense/tts_chunk` amplitude drives mouth shape

### Phase 4 — performance polish

* Metal-backed Canvas where it matters
* Window can be transparent for OBS capture
* Settings panel (font, background colour, asset roots)

### Phase 5+ — visualisation surfaces (future)

* Skill tree sidebar (reads `/sense/xp_awarded` + skill_tree.json)
* Personality character sheet (reads identity.yaml + personality.json)
* Performance picker (list of `<instance>/timelines/*.json` to play)

## What about the 0.3.0 Swift work at `apps/JaegerOS/`?

Survey: `apps/JaegerOS/` has `.build/` artifacts from the walked-
back 0.3.0 daemon-attached Swift app (DaemonAPI, ChatView,
PillPanel, etc.).  The source files were removed in 0.3.0 cleanup;
only build artifacts remain.

**Decision: don't reuse.**  That code was for a different
architecture (daemon-attached chat client + always-on pill).  The
new app is rendering-focused.  Starting clean keeps the design
crisp.

Cleanup task: delete `apps/JaegerOS/.build/` so the directory tree
stays tidy — 0.5.x followup.

## Honest scope note (matches my earlier ack)

I can scaffold + write the JSON ↔ animation event handler + a basic
SwiftUI window.  Production-quality Mac SwiftUI work has real depth
beyond what one autonomous session produces.  The Phase-1 scaffold
in this commit is FUNCTIONAL — opens a window, connects to a
WebSocket, displays incoming frames as raw pixels — but not as
POLISHED as a dedicated Mac dev would produce.  Iteration through
Phases 2-5 is where the polish happens.

the contract testable + swappable.  If we ever want a browser
renderer instead, we write one and the Python side doesn't notice.
