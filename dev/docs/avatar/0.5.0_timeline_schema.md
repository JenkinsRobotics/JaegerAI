# 0.5.0 Timeline schema — multi-track, OTIO-inspired

**Status:** design doc, scaffold pending
**Operator-locked 2026-06-08:** "don't be locked down to mscript;
reference industry-standard formats."
**Inspiration:** OpenTimelineIO (Pixar / ASWF), Lottie (Adobe), EDL

## Why multi-track

Mscript is single-track sequential.  An agent performance —
"greeting" — needs:

```
animation track:  [neutral face]──[smile]────[wave]──────[neutral]
speech track:     ───[Hi, how are you today?]────────────────────
sound track:      [chime]──────────────────────────────[chime]
motion track:     ─────────────[raise hand]──[lower]─────────────
                                                  (deferred to 0.6)
```

These run in parallel.  Mscript can't express that cleanly.  A
multi-track timeline can — each track has independent events at
absolute time offsets, all played in lockstep.

## Industry references (informing — not copying)

* **OpenTimelineIO (OTIO)** — Pixar/ASWF.  JSON tracks with clips,
  transitions, markers, effects.  Standard in VFX.
* **Lottie** — Adobe After Effects JSON.  Vector animation +
  keyframes.  Used everywhere mobile + web (animated icons).
* **EDL** — broadcast editorial.  Linear, simple, decades old.
* **Spine JSON** — sprite-rig animation.  Keyframed parameter
  curves.

JROS borrows OTIO's multi-track + clip-with-time-range model.
Lottie's per-track keyframes inform future L5 (rigged) and L6
(generative) authoring.  Mscript becomes a one-way compiler input.

## Schema (msgspec.Struct + JSON-serialisable)

```python
class TimelineClip(msgspec.Struct):
    """One event/asset on a track.

    For animation tracks: clip is a play-an-asset event.
    For speech tracks:    clip is a say-this-text event.
    For sound tracks:     clip is a play-this-sound-file event.
    For motion tracks:    clip is a motion-command (0.6+).
    For light tracks:     clip is a light-command (0.6+).
    """
    t_offset_ms: int                # start time, ms from track start
    duration_ms: int                # 0 = natural duration (asset-defined)
    payload: dict                   # track-kind-specific shape
                                    #   animation: {adapter, asset, params}
                                    #   speech:    {text, voice}
                                    #   sound:     {path, volume}
                                    #   motion:    {linear_x, angular_z, ...}
                                    #   light:     {rgb, pattern}
    label: str = ""                 # operator-visible name (optional)

class TimelineTrack(msgspec.Struct):
    """A homogeneous sequence of clips for one consumer node."""
    kind: str                       # "animation"|"speech"|"sound"|"motion"|"light"
    clips: list[TimelineClip] = []

class Timeline(msgspec.Struct):
    """The full multi-track sequence."""
    name: str                       # "greeting" / "goodbye" / "thinking-pose"
    description: str = ""
    duration_ms: int = 0            # total duration; 0 = max(track end times)
    loop: bool = False
    tracks: list[TimelineTrack] = []
    schema_version: int = 1
```

## Example JSON — "greeting" performance

```json
{
  "name": "greeting",
  "description": "Three-second hello with a smile + spoken welcome.",
  "duration_ms": 3000,
  "loop": false,
  "schema_version": 1,
  "tracks": [
    {
      "kind": "animation",
      "clips": [
        {"t_offset_ms": 0, "duration_ms": 500,
         "payload": {"adapter": "image",
                     "asset": "faces/neutral.png"},
         "label": "neutral"},
        {"t_offset_ms": 500, "duration_ms": 1500,
         "payload": {"adapter": "sprite",
                     "asset": "faces/smile.json",
                     "params": {"loop": false}},
         "label": "smile"},
        {"t_offset_ms": 2000, "duration_ms": 1000,
         "payload": {"adapter": "image",
                     "asset": "faces/neutral.png"},
         "label": "back to neutral"}
      ]
    },
    {
      "kind": "speech",
      "clips": [
        {"t_offset_ms": 200, "duration_ms": 2200,
         "payload": {"text": "Hello — good to see you."},
         "label": "greeting line"}
      ]
    },
    {
      "kind": "sound",
      "clips": [
        {"t_offset_ms": 0, "duration_ms": 200,
         "payload": {"path": "chimes/welcome.wav",
                     "volume": 0.6},
         "label": "welcome chime"}
      ]
    }
  ]
}
```

## Runtime — the timeline runner

```python
class TimelineRunner:
    """Schedules clips from a Timeline onto the bus at their
    absolute offsets.  One runner per active timeline.

    On start: sorts all clips across all tracks by t_offset_ms;
    builds a wall-clock schedule; publishes per-clip events at the
    right moments.

    Per-track publishing — kind dispatches to the right topic:
      animation → /act/animation     (AnimationCommand)
      speech    → /act/speech        (SpeechCommand)
      sound     → /act/audio_out     (or a sound-specific topic)
      motion    → /act/motion        (deferred)
      light     → /act/light         (deferred)

    Publishes /sense/timeline_progress at:
      - start of timeline
      - per-clip events (optional, configurable)
      - end of timeline (natural or interrupted)
    """
```

## Mscript → Timeline compiler

```
mscript                          Timeline (compiler output)
─────────────────────────────    ─────────────────────────────
MATH K[1] D[61.0] FG[255,0,0]    animation track:
WAIT D[0.1]                        clip @0ms: math adapter,
MATH K[1] EVENT[start] DUR_M[1]              asset = K[1], 100ms
WAIT D[60.9]                       clip @100ms: math adapter
                                              with EVENT={start}
                                              duration 60_900ms
```

* Single-track: mscript only has one rendering source at a time.
* WAIT becomes empty time between clips.
* K[id] references resolve to asset paths from the [Resources]
  section.
* EVENT[name] becomes ``payload.event`` on the clip.
* Errors during compilation are surfaced; we don't silently
  half-load broken mscript.

Existing `.mscript` files keep working via this compiler.  No new
mscript is authored.

## Authoring workflow

1. Operator writes Timeline JSON by hand (initially), or uses a
   future visual editor in the Swift app.
2. Saves to `<instance>/timelines/<name>.json`.
3. Agent tool `play_timeline(name)` publishes a
   :class:`TimelineCommand` with the name.
4. TimelineRunner loads the JSON, schedules.

Inline timelines — agent can compose a Timeline ad-hoc, serialise
to JSON, and pass via `TimelineCommand.timeline_json` for one-shot
performances.

## Implementation order

```
1. jaeger_os/timeline/schema.py         msgspec.Struct types
2. jaeger_os/timeline/runner.py         Wall-clock scheduler
3. jaeger_os/timeline/mscript_compile.py  one-way compiler
4. jaeger_os/timeline/loader.py         load from instance dir
5. Per-instance:
     <instance>/timelines/*.json        operator-authored timelines
     <instance>/timelines/legacy/*.mscript  imported from Mochi
6. Agent tool: play_timeline(name) / compose_timeline(json)
```

Tests cover: schema round-trip, runner timing, mscript-compile
parity for the Mochi demo scripts, interrupt handling.

## Open questions for operator (non-blocking)

1. Do we want **transitions** (fade between clips) in 0.5.0, or
   defer?  My lean: defer.  L1-L4 adapters don't have native fade
   support; faking it is more complex than worth.
2. Do we want a **markers** track (event labels with no payload,
   for debugging/visualisation)?  My lean: yes, but post-0.5.0.
3. Does the **agent need a tool** to AUTHOR timelines (compose
   from a description), or just to PLAY them?  My lean: play
   first; author later when the visual editor lands.
