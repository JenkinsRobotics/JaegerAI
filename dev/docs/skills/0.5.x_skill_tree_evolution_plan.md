# Skill tree — evolution plan

**Status:** queued (planning doc; no code lands from this commit)
**Date:** 2026-06-09
**Companion:** [`SKILL_TREE.md`](SKILL_TREE.md) (the foundation contract)

The honest framing the operator pinned 2026-06-09:

> *Today, the skill tree is a registry + log with structural
> cascades.  It hasn't yet earned the name "skill tree" in any
> behavioural sense.  For it to BECOME load-bearing, it needs
> to affect what the agent says, what it does, or how it does it.*

This doc names the four directions it could evolve, picks two as
the recommended starting point, and frames concrete first commits
for each.  Out of scope for this doc: actually doing them — each
mode below is its own future PR.

## What the skill tree IS today

```
SkillNode        = {id, name, category, level, xp, xp_to_mastery,
                    prerequisites, unlocks, status}
SkillTree        = a DAG of SkillNodes (msgspec, JSON-persisted)
SkillTreeRegistry = thread-safe registry + atomic save +
                    level-up cascade + listener API
XpEmitter        = bus subscriber: /sense/xp_awarded → registry
seed.py          = pre-populates default catalog (26 skills)
```

What it does at runtime today: tracks numbers, cascades
transitions, publishes bus events.  What it does to the agent's
BEHAVIOUR today: nothing.  No tool gates on it.  The brain
doesn't read it.  Voice gate doesn't consult it.

Adjacent infrastructure that exists but isn't wired:
- `XpEmitter` subscribes to `/sense/xp_awarded` — but no
  publisher fires events yet.
- `seed_default_tree(registry)` registers 26 skills — but the
  registry is empty until called.

This is foundation.  Not yet a load-bearing system.

## Four directions the tree could evolve

### Mode 1 — Prompt feedback (self-awareness loop)

The brain reads a compose block in its system prompt every turn:

    ## What I've gotten good at
      ★ MASTERED  animation.image    (520 successful uses)
      ★ MASTERED  voice.tts          (1200 successful uses)
      ● ACTIVE    animation.sprite   L2 (in progress)
      ○ AVAILABLE animation.gif      (sprite mastery unlocked this)
      ○ AVAILABLE voice.lip_sync     (tts + math unlocked this)

The brain KNOWS its tree shape and can reason about it.  It might
say things like *"I've gotten good at images — I should try
sprites now."*  Mirrors how the personality compose block landed.

**First commit:** `compose_skill_block(registry) → str` in
`skill_tree/compose.py` + append in `assemble_prompt` after the
personality block.  Tests pin the format the same way personality
band wording is pinned.  Effort: ~half a session.

### Mode 2 — Capability gating

Lock tools behind skill status:

    @requires_skill("animation.gif", status="available")
    def play_gif(...): ...

The brain literally can't reach `play_gif` until prereqs are
mastered.

**Recommendation: SKIP for JROS.**  The operator's instinct is
"agents start usable, not crippled."  Gating L3+ skills behind
prereq mastery is RPG-clean but artificial here.  Don't build it
unless a concrete capability needs to be gated for safety
reasons (and at that point, safety/permissions is the right
layer, not skill_tree).

### Mode 3 — Behavior shaping (the most interesting)

Skill level changes EXECUTION QUALITY, not whether the tool exists.

    voice.tts  L1  → default Kokoro params, neutral cadence
               L2  → emphasis cues honoured
               L3  → speech_patterns from personality applied
               L4  → mid-sentence prosody adjustments

    animation.image  L1  → static frame, default 1 second hold
                     L2  → entry/exit fades
                     L3  → ken-burns subtle zoom

Same tool surface; level decides HOW it runs.  Levels = competence
increments.  This is the videogame-character match — at L5 your
character isn't NEW, they DO existing things better.

**First commit:** Pick ONE skill (probably `animation.image`
since it's the simplest) and wire level-conditional behaviour.
ImageAdapter reads `registry.get("animation.image").level` and
chooses between three render paths.  Tests pin the per-level
output shape.  Effort: ~half a session per skill.

This is incremental: each skill that gets wired is one small
commit.  Over weeks/months the system gradually goes from
"counter" to "shape."

### Mode 4 — Identity surface (visualisation)

The tree IS the agent's character sheet — what they've experienced,
what they've mastered, what they could become.

  - `jaeger skills` (already shipped) — terminal-facing
  - Swift app radial-graph tab — the videogame-aesthetic UI from
    the reference images
  - Two months from now: Lilith has `voice.lip_sync` +
    `animation.math` mastered (avatar-first instance); Jarvis has
    `core.web_search` + `core.memory` mastered (research-first
    instance).  Their tree SHAPES differ from each other.

**First commit:** Swift `SkillTreeView.swift` reads
`<instance>/progression/skill_tree.json` and renders the radial
graph.  Effort: ~one session of focused Swift work.

No new Python code — just a viewer over data Mode 1 + 3 already
populate.

## Recommended sequence

The smallest move that earns the name "skill tree":

1. **Now → next week — XP emission pass.**  Nothing flows today.
   Wire `award_xp(bus, skill_id, 1)` into tool dispatch
   on success + a `_TOOL_TO_SKILL` mapping.  Without this, every
   later mode is decorative.
   - Effort: 1-2 hours
   - Yield: `./jaeger skills` finally shows progression after a
     turn.

2. **Mode 1 (prompt feedback).**  Brain sees its tree state every
   turn.  Single biggest impact for the smallest code.
   - Effort: half a session
   - Yield: brain self-awareness; operator visibly sees the
     skill state affect the brain's framing.

3. **Mode 4 (Swift visualisation).**  Read-only viewer, no new
   data.
   - Effort: one session of focused Swift work
   - Yield: the videogame surface that matched your reference
     images.

4. **Mode 3 (behaviour shaping) — incremental.**  Pick ONE skill
   per commit and wire its level-conditional behaviour.  Start
   with `animation.image` because it's the simplest + most
   visible.
   - Effort: half a session per skill
   - Yield: the long-burn payoff — competence visibly grows.

Total to a meaningfully load-bearing skill tree: ~3 sessions of
focused work, spread over 0.5.x → 0.6.

## Explicitly deferred / out of scope

- **Mode 2 (gating).**  Don't.  Use `core/safety/permissions/`
  for genuine safety gates.  Skill levels should shape execution
  quality, not artificially block capabilities.

- **Reward signal for fine-tuning.**  Some future possibility:
  high-XP traces become preferred examples for a fine-tune run.
  Real but ≥6 months out; needs its own design pass when local
  fine-tuning is set up.

- **Cross-instance skill transfer.**  If instance A has
  `animation.math` mastered and you fork instance B from A,
  should B inherit?  Genuine question.  Defer until we have a
  second instance worth caring about.

- **Skill regression.**  XP can only go UP today.  Should an
  unused skill DECAY over time?  Probably yes for realism
  (forgetting is real), but not in any near horizon.
