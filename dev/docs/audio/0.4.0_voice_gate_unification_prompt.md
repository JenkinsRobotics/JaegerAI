# Implementation request — voice gate unification + TUI split-screen

This consolidates two operator-locked architectural calls made on
2026-06-07 after testing Codex's first gating fix:

1. **Single-pass gate** (VoiceLLM-style), NOT the two-call gate
   Codex landed.  Latency matters; the deterministic filters
   we already have ARE the "first anti-junk agent" — adding a
   dedicated LLM gate call makes it three agents when it should
   be two.
2. **TUI vertical split** — conversation pane on top, rolling
   voice-activity pane on the bottom, status bar at the very
   bottom.  Voice activity is a BOUNDED rolling window — never
   expands; old events scroll out.

Both land on the `0.4.0` branch before the eventual main merge.

## Architectural model (operator-locked)

```
                    ┌──────────────────────────────┐
microphone  ──────► │ Stage 1: deterministic       │
                    │ filters (the "first agent")  │
                    │                              │
                    │  - non-speech marker filter  │
                    │  - self-speech filter (0.75) │
                    │  - duplicate suppression     │
                    │  - minimum content gate      │
                    │  (lives in AudioSession)     │
                    └─────────────┬────────────────┘
                                  │
                                  │ /sense/transcript
                                  ▼
                    ┌──────────────────────────────┐
                    │ Stage 2: primary LLM agent   │
                    │ (the "second agent")          │
                    │                              │
                    │  System prompt teaches:       │
                    │  - "much of what you hear     │
                    │     is NOT addressed to you" │
                    │  - "begin EVERY reply with    │
                    │     exactly one tag:          │
                    │     <ignore> or <reply>"     │
                    │  - explicit junk examples     │
                    │                              │
                    │  Streaming response:         │
                    │  - peek first ~30 chars      │
                    │  - <ignore> → abort,         │
                    │     don't dispatch to TTS    │
                    │  - <reply>  → strip tag,     │
                    │     stream rest to TTS       │
                    └──────────────────────────────┘
```

Two stages.  No third dedicated gate LLM call.  Same latency as
a normal agent turn (the gate decision is "free" because it
happens DURING streaming, not before it).

* `jaeger_os/core/voice/input_gate.py` defines
  `classify_voice_input()` that makes a dedicated LLM call with
  `max_tokens=8`.
* TUI calls this before forwarding to the agent.

What to do:

* **Keep** `VOICE_INPUT_GATE_PROMPT` (the system prompt body
  with the strong example list).  Its anti-junk training
  signal is good.  Move the prompt text into a constant in a
  prompt module (e.g.,

  `classify_voice_input(...)`.  Replace with the single-pass
  parse in G2.
* **Keep** the deterministic filters in
  `jaeger_os/core/voice/non_speech.py`,
  `jaeger_os/core/voice/llm_gate.py` (the `<ignore>`/`<reply>`
  parser), and the self-speech filter material in
  voice_loop.  These ARE the first agent.

If repurposing is cleaner as one commit and the deletion as a
second commit, do both.

## G2 — single-pass streaming prefix-peek

The primary agent's normal LLM turn now does the gating.  No
separate call.

Implementation:

* Pick the unified turn-running function (the one shared by
  TUI + voice_loop after the audio refactor at commit
  `5602339`).  In voice mode, BEFORE the response streams out
  to the TUI / TTS:
  - Buffer the first **30 characters** of the streamed response
    (constant: `LLM_GATE_BUFFER_CHARS = 30`).
  - Case-insensitive search for `<ignore>` or `<reply>` within
    that buffer.
  - If `<ignore>` found first → CANCEL the generation
    (`abort` / `stop_event` / whatever llama-cpp exposes),
    don't dispatch ANY of the response to TTS, log "[gate:
    ignored]" to the voice-activity pane.
  - If `<reply>` found first → strip the tag prefix, dispatch
    the rest of the streaming response normally.
  - If neither tag in the first 30 chars (model didn't follow
    instructions): default to `<reply>` (be forgiving), log
    "[gate: fallback reply]" for debugging.

VoiceLLM reference: `core/runners/orchestrator.py:279-311` —
mirror the buffer-then-parse logic, but apply to JROS's
streaming response model.

JROS's `LlamaCppPythonClient` and `MlxClient` both already
stream; the change is in the consumer that reads the stream.

Tests:

* Mock LLM emitting `<ignore>` as first 8 chars → response
  cancelled, no TTS dispatch, voice-activity log shows
  "gate: ignored."
* Mock LLM emitting `<reply>foo` → response continues, TTS gets
  "foo", voice-activity log shows "gate: accepted."
* Mock LLM emitting `hello world without tag` → fallback to
  reply, TTS gets full text.
* Streaming abort actually stops generation (llama-cpp's `stop`
  mechanism works; verify with timing assertion).

## G3 — system prompt enrichment

The primary agent's system prompt teaches the gate behavior.
VoiceLLM does this at `config.py:35-60`; JROS already has
`core/prompts/rules.py` with similar but weaker training.

What to do:

* Audit `core/prompts/rules.py` for the voice gate rule block.
* Strengthen with VoiceLLM-style framing:
  - Explicit acknowledgement that "you receive transcriptions
    from an always-on microphone, so much of what you hear is
    NOT directed at you — ambient speech, transcription
    artifacts, single-word fragments, conversations between
    other people, keystroke noise."
  - Mandate: "Begin EVERY reply with exactly one tag:
    `<ignore>` (input is not addressed to you, output the tag
    and nothing else) OR `<reply>` (input is addressed to you,
    respond after the tag)."
  - Default-to-`<ignore>` framing.
  - Examples list (use the strong list already in Codex's
    `VOICE_INPUT_GATE_PROMPT` — movie quotes, ad copy, song
    lyrics, ambient fragments).
* Only injected when `JAEGER_VOICE_GATE=1` (already the
  pattern) so non-voice agent turns aren't affected.

The system prompt must teach the model to emit `<ignore>` /
`<reply>` BEFORE any tool call dispatch.  If the agent's tool
loop fires a tool call before the gate decision is made, the
gate doesn't work.  Worth a dedicated test:

* Mock LLM trying to emit `text_to_speech(...)` tool call before
  `<reply>` tag → test asserts this case is handled (either
  reject + retry, or force the tag injection at the streaming
  layer).  Operator-flagged risk area.

## G4 — TUI vertical split with bounded rolling voice activity

Layout:

```
┌─────────────────────────────────────────────────────┐
│  💬 Conversation                            ~70%    │
│                                                     │
│   ✦ jarvis                                          │
│   <agent reply, possibly multi-paragraph>           │
│                                                     │
│   ❯ <user turn>                                     │
│                                                     │
│   (scrollable; never truncated)                     │
├─────────────────────────────────────────────────────┤
│  🎙 Voice Activity                          ~20%    │
│   [last N events; bounded; rolling]                 │
│                                                     │
│   [heard] 'Hey Jarvis, what time is it?'           │
│   🎙 voice gate: accepted                            │
│   [skipped — non-speech: '[BLANK_AUDIO]']          │
│   [heard] '(footsteps)' → 🤫 ignored               │
├─────────────────────────────────────────────────────┤
│  ✦ gemma-12B │ ctx 14.6K/32.8K │ 🎙 on │ ⏲ 1m 24s  │
└─────────────────────────────────────────────────────┘                                   ~10%
```

Implementation:

* Use `rich.live.Live` + `rich.layout.Layout` for the redraw
  loop.
* Three regions:
  - `conversation` (ratio=7) — scrollable history; uses
    existing `_render_agent_reply` / banner / boot-panel code
    paths.
  - `voice_activity` (ratio=2) — bounded `collections.deque`
    of last ~10 events (operator-tunable via config:
    `voice.activity_log_lines`, default 10).  Old events drop
    off as new ones arrive — NEVER expands the window.
  - `status` (ratio=1) — existing status bar; same content.
* Routing layer: introduce a small helper `_log_voice(...)` and
  `_log_conversation(...)` that route messages to the right
  region.  Replace the existing
  `self.console.print(f"[dim]🤫 voice gate: ignored...")`
  call sites with `self._log_voice(...)`.

Voice-activity entries that should land in the rolling pane:

* `[heard] '<text>'` — STT phrase appeared
* `[skipped — non-speech: '...']` — non-speech marker filter hit
* `🎙 voice gate: accepted` / `🤫 voice gate: ignored (<reason>)`
* `🎙 coalesced — will handle latest voice batch next.`
* `🤫 self-speech filter dropped (sim=0.91)`

Conversation entries that stay in the main pane:

* User-typed messages
* `✦ jarvis` agent reply panels
* `/<slash command>` echoes
* `[boot]` lines + boot panel
* Error messages

Bonus polish (worth doing while you're in there):

* **Aggregate consecutive non-speech skips.**  Instead of 5
  lines of `[skipped — non-speech: '[BLANK_AUDIO]']`, render
  one line: `[skipped — 5 non-speech events]`.  Operator
  explicitly mentioned "i dont need to see a long continouse
  skipped list."
* **Color-code the voice activity:**
  - dim gray for skipped non-speech (visible but background)
  - white for `[heard]` lines (real STT firings)
  - green for `🎙 accepted`
  - yellow for `🤫 ignored`
  - red for self-speech-filter drops + gate errors

Tests:

* `_log_voice` writes route to voice region only; never to
  conversation.
* Voice activity deque caps at the configured length; oldest
  drops off.
* Aggregation: 3+ consecutive non-speech events render as
  one aggregate line, not 3.
* Layout resize: layout still renders sanely at 80-col and
  200-col widths.

## G5 — addressed_hint context injection during follow-up window

Currently the gate prompt is the same regardless of
conversation state.  VoiceLLM injects an extra clause during
the follow-up window:

> "This transcript is inside an active conversation or appears
> related to the current topic.  Treat it as addressed to you
> unless it is clearly ambient speech."

Implementation:

* The `AudioSession` knows whether we're inside the
  follow-up window (already tracked at
  `voice_loop.py:573` via `_in_active_followup`).
* When publishing `/sense/transcript`, include a boolean
  `active_followup: bool` field on the topic.
* The agent loop, when assembling the system prompt for the
  turn, conditionally appends the addressed_hint clause when
  `active_followup=True`.
* Strict default (idle): default to `<ignore>` for borderline
  cases.
* Permissive default (follow-up): default to `<reply>` for
  borderline cases.

Add `active_followup` to `Transcript` msgspec.Struct schema in
`topics.py`.  This is a small SSOT addition; document in the
schema's class docstring.

Tests:

* Borderline phrase WITHOUT follow-up window → LLM gates
  `<ignore>`.
* Same phrase WITH follow-up window active → LLM gates
  `<reply>`.
