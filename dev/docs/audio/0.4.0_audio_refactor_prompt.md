## Three design calls already made

### Design call 1 — rename, NOT parallel

The plan suggested keeping `STTNode` as a "narrow adapter for
tests and simple pipelines" and adding a new `AudioSessionNode`
alongside it.  Implementing agent pushed back; operator agreed.

**Decision: rename and expand `STTNode` in place.**

The current `STTNode` is already a thin wrapper around
`WhisperSTTContinuous`.  Expanding its responsibilities to
include AEC + filters + follow-up state + barge-in detection is
a contained change.  Two nodes-that-do-similar-things would be
a maintenance pothole.

Practical implementation:

* Rename `jaeger_os/nodes/stt/` → `jaeger_os/nodes/audio_session/`.
* Rename `STTNode` class → `AudioSessionNode`.
* Update every import + test reference.
* Keep `STTNode` as a back-compat alias for one release if doing
  so is trivial; otherwise hard-rename and update tests.  Either
  is fine — judgment call.

### Design call 2 — AEC reference buffer coupling

The plan said:

> In monolithic mode, share this reference in process.  In
> multiprocess mode, add a lightweight reference-feed topic only
> if it is actually needed.

Implementing agent flagged this as hand-waving.  We need a real
answer.

**Decision: in monolithic mode, the `AudioSessionNode` reads
the reference buffer via `runtime.get_synth().reference_buffer`
(or equivalent direct accessor).  Document this as a known
monolithic-only coupling.  In multiprocess mode (future Track
A.7 + B.4 work), introduce `/sense/tts_reference` binary topic
then.**

This is a deliberate compromise: cross-node refs in monolithic
mode are an architectural smell, but the alternative (raw audio
on the bus right now) is worse.  Acknowledge it explicitly in:

* The `AudioSessionNode` class docstring
* `dev/docs/ROADMAP_0.4.md` open questions

So future readers know it's a known wart pending the
multiprocess Bus work, not an oversight.

### Design call 3 — performance budgets need measurement

The plan listed budgets like "transcript publish after STT
finalization: under 20 ms local overhead."  These are good
targets but currently aspirational.

**Decision: add at least ONE bench case that actually measures
the realtime path** — pick the easiest one (e.g. mock-STT
finalises a phrase, measure wall time until `/sense/transcript`
fires).  Doesn't need to be a polished bench; a `time.perf_counter()`
diff + a print is fine as the first cut.

Don't claim Step 4 done until at least one budget has a
measurement, even if it's a smoke-quality one.
