# Voice gating — VoiceLLM vs. JROS comparative review

**Status:** 0.4.0 release-blocker review
**Date:** 2026-06-07
**Trigger:** operator-reported failure — JROS treats TV/movie/
background fragments as real user turns; VoiceLLM doesn't.
**Reviewer:** implementing agent (cross-reading
[`/Users/jonathanjenkins/GITHUB/VoiceLLM/`](file:///Users/jonathanjenkins/GITHUB/VoiceLLM)
against JROS current state including Codex agent's uncommitted
[`input_gate.py`](../jaeger_os/core/voice/input_gate.py))

## What's actually wrong

Two real problems, related but distinct:

### Problem A — the architectural rot

> *"i dont want different modes hidden in different interfaces..
> it should always be the same pipeline irrelevant of the current
> interface"* — operator, 2026-06-07

JROS has **TWO voice pipelines** that diverged:

| | Standalone (`voice_loop.py`) | TUI (`voice_session.py`) |
|---|---|---|
| STT instance ownership | bus-shared via STT node | (was) direct ownership → now AudioSessionNode after refactor |
| Self-speech filter | implemented in-loop at `voice_loop.py:551-571` | implemented separately or not at all |
| LLM gate parsing | `parse_gate(text)` after agent reply at `voice_loop.py:~580` | (Codex's fix) pre-agent classifier at `input_gate.py` |
| Active follow-up retry | yes, conditional on window | n/a |
| Non-speech marker filter | via `non_speech.py` import | via `non_speech.py` import (good — shared) |
| Pending-phrase queue | configurable | n/a |

That's the rot: **the same conceptual operation lives in different
files with different parameters and different orderings.**  TV
audio passing voice_loop's filters might fail TUI's gate, and
vice-versa.  The user experience depends on which entry point you
came in through — exactly what the operator says they hate.

### Problem B — JROS's filter chain is shorter than VoiceLLM's

Even on the paths that ARE filtering, JROS has FEWER gates than
VoiceLLM does:

```
VoiceLLM pipeline (every phrase passes ALL of these):
  1. WebRTC VAD (aggressiveness=2)              ← acoustic frame-level gate
  2. Energy threshold (RMS > 0.008)             ← second acoustic gate
  3. Whisper internal confidence                ← model gate
  4. Duplicate suppression (>92% similarity)    ← STT-rolling gate
  5. Self-echo filter (>75% sim to last reply)  ← orchestrator gate
  6. LLM <ignore>/<reply> tag                   ← semantic gate
  7. Addressed_hint context (follow-up window)  ← contextual gate

JROS pipeline today (best path = voice_loop):
  1. (energy-mode only) energy threshold         ← partial — two_pass lacks this
  2. Whisper internal confidence                 ← model gate
  3. Non-speech marker filter ([BLANK_AUDIO]…)   ← deterministic
  4. (when on) self-speech filter — 0.75 default ← parity with VoiceLLM
  5. LLM <ignore>/<reply> tag                    ← semantic gate
  6. (codex fix, TUI only, uncommitted)
     PRE-LLM input_gate.py classifier            ← new dedicated gate
```

JROS is missing:
- **WebRTC VAD** (we have energy-only or Whisper-internal VAD)
- **Duplicate suppression at STT-rolling level**
- **Addressed_hint context injection during follow-up window**

And the items we DO have aren't applied uniformly across both
interfaces.

## What makes VoiceLLM resistant — the WHY

Three design choices, not one.  Important distinction because
porting any single one in isolation won't close the gap.

### 1. Layered deterministic gates BEFORE the LLM ever sees a phrase

By the time VoiceLLM's LLM is asked "is this addressed to you?",
five rule-based gates have already rejected:
- silence (WebRTC VAD)
- click/tap/plosive (energy floor)
- low-confidence STT (Whisper threshold)
- repeated transcriptions (duplicate suppression — 92% similarity)
- self-echo (75% similarity to last reply)

So the LLM only sees phrases that are at minimum: voice-ish,
loud enough, intelligible, not a duplicate, and not the agent
talking to itself.  TV/movie fragments still pass most of these
(they're voice-ish, loud enough, intelligible) but the LLM gate
catches them at step 6.

JROS today skips several of these and exposes the LLM gate to
more noise.

### 2. The LLM gate prompt is anti-junk-trained

VoiceLLM's system prompt at `config.py:35-60` doesn't just say
"prefix with `<ignore>` or `<reply>`."  It teaches the domain:

> "You receive transcriptions from an always-on microphone, so
> much of what you hear is NOT directed at you — ambient speech,
> transcription artifacts, single-word fragments, conversations
> between other people, keystroke noise."

Then it includes explicit examples (`[BLANK_AUDIO]` → `<ignore>`,
"test" → `<ignore>`, single isolated words → `<ignore>`).
**Default is `<ignore>`; the agent must justify `<reply>`.**

This means a confused Gemma defaults safe (silent) instead of
defaulting eager (responding to TV).

JROS's gate prompt is structurally similar but should be
audited for the SAME conservative training signal.  Codex's
`input_gate.py` prompt is actually well-shaped on this front —
the examples list is strong (movie quotes, ad copy, song lyrics,
etc.).

### 3. The addressed_hint context inside the follow-up window

`plugins/llm_core/node.py:93-103` — during the 15s follow-up
window after a reply, VoiceLLM injects context into the LLM call:

> "This transcript is inside an active conversation or appears
> related to the current topic.  Treat it as addressed to you
> unless it is clearly ambient speech."

This dual-mode behavior — STRICT during cold start (default
`<ignore>`), PERMISSIVE during follow-up (default `<reply>` if
plausibly related) — is the reason VoiceLLM feels responsive
during conversation but doesn't fire on background noise when
idle.

JROS currently has the same prompt regardless of state.  This is
a real gap.

## Codex agent's fix — what it does + the architectural call

The uncommitted change adds `jaeger_os/core/voice/input_gate.py`:

* New `classify_voice_input(client, transcript, *, active_followup)`
  function
* Calls the LLM with `max_tokens=8` to classify accept/reject
* Uses a dedicated system prompt with strong examples
* Conditionally includes a follow-up permissiveness hint
* Returns `VoiceInputGateDecision(accepted, reason, raw)`

### The architectural call this represents

VoiceLLM uses a **single-pass gate**: the LLM's normal turn
response starts with `<ignore>` or `<reply>` as the first 30
chars; if `<ignore>` found early, the rest is dropped.  One LLM
call per phrase.

Codex's fix uses a **two-call gate**: a cheap dedicated LLM
classification call (~8 tokens) first; on accept, a full agent
turn follows.

|  | Single-pass (VoiceLLM) | Two-call (Codex) |
|---|---|---|
| LLM calls per accepted phrase | 1 | 2 |
| LLM calls per rejected phrase | 1 (short) | 1 (very short, ~8 tok) |
| Gate prompt isolated from agent context | ✗ tangled | ✓ clean |
| Gate decision logged separately | ✗ in the response | ✓ explicit |
| Tool/skill access during gate | yes (could cause weird states) | no (clean) |
| Cost on accepted turn | ~normal | +50-200ms gate latency |

**Both are valid.**  Codex's design is the cleaner separation
of concerns; VoiceLLM's is more efficient at scale.  For 0.4.0
the question is "does the gating work?" not "is it optimal" — so
either approach can land.

The bigger problem isn't single-pass vs two-call.  **It's that
the gate only runs on the TUI path.**  Standalone voice_loop
still uses the single-pass approach (parses tags out of the
agent reply post-hoc).  Two pipelines, one architectural
philosophy in each.

## What needs to land before 0.4.0 main merge

In priority order:

### MUST — unify the pipeline through one entry point

Steps 1-4 of the audio refactor (already authorized at
[`0.4.0_audio_refactor_prompt.md`](0.4.0_audio_refactor_prompt.md))
are the architectural prerequisite for this whole fix.  Until
both voice_loop AND voice_session consume `/sense/transcript`
from the same AudioSessionNode, you can't have ONE gate
implementation; you'll always have two paths to keep in sync.

This is the non-negotiable foundation.

### MUST — move the gate into the unified path

The gate logic (whichever flavor we pick — Codex's or
single-pass) lives at the `/sense/transcript` consumer layer,
not in `voice_session.py` and not in `voice_loop.py`.

Both paths are then thin wrappers that:
1. Render UI / log "[ignored]" status to operator
2. Hand accepted transcripts to the SAME turn-running function

`jaeger_os/core/voice/input_gate.py` already exists; it just
needs to be the gate that BOTH interfaces consult, not just TUI.

### MUST — pick gate philosophy + commit to it

Two paths:

**Option A:** Keep Codex's two-call gate.  Pros: cleaner code,
clear logging of why something was ignored.  Cons: adds gate
LLM latency to every accepted turn.

**Option B:** Port VoiceLLM's single-pass gate.  Pros: matches
the reference exactly; one LLM call per phrase.  Cons: gate
decision is tangled with the response generation.

My lean: **Option A**, keep Codex's gate.  Reasons:
- It's already working and committed to working in the operator's
  test plan
- The gate latency is ~100-200ms; agent turn latency is already
  in seconds; the relative cost is small
- The cleanliness of "gate is its own concern, not the agent's
  problem" is worth it long-term, especially when tools/skills
  start to matter
- Single-pass tangles with tool dispatch in ways that get ugly
  (what happens if the agent calls a tool and THEN emits
  `<ignore>`?)

### MUST — close the JROS-specific filter gaps

Even with the unified gate, add the deterministic pre-gate
filters JROS is missing:

1. **Minimum content length gate** — drop phrases under 3
   characters or under 2 syllables.  Catches "uh", "ah",
   single-word noise.
2. **Duplicate suppression at the AudioSession level** — track
   the last N transcripts (N=5); drop new ones with >0.92
   similarity.  Catches babble loops.
3. **Self-speech filter ON by default** — currently default
   OFF.  Set default ON, threshold 0.75 (VoiceLLM parity).

All three are cheap, deterministic, and reject obvious
junk before any LLM call.

### SHOULD — match VoiceLLM's anti-junk system prompt training

Audit JROS's `VOICE_LLM_GATE_RULE` (in
`core/prompts/rules.py` or similar) against VoiceLLM's
`config.py:35-60` for:
- "default to `<ignore>`" framing
- Explicit junk examples
- "most audio is not addressed to you" priming

Codex's `VOICE_INPUT_GATE_PROMPT` is well-shaped already; the
agent's main-conversation gate rule should match its
conservative tone.

### SHOULD — addressed_hint context during follow-up window

The single nuanced VoiceLLM trick we don't yet have.
Implement: when `active_followup=True` AND we're inside the
follow-up window, the gate prompt gains a clause:

> "This transcript is inside an active conversation or appears
> related to the current topic.  Treat it as addressed to you
> unless it is clearly ambient speech."

Codex's `input_gate.py` already has the `followup_hint` plumbing
(lines 71-75) — confirm it's wired through the active-followup
state correctly from both interfaces.

### COULD — defer to 0.5 (don't gate 0.4.0 on these)

* **WebRTC VAD** — real upgrade over energy/Whisper-internal VAD
  but requires WhisperSTT refactor.  0.5 Track G or H.
* **STT-rolling duplicate suppression** — VoiceLLM has it; JROS
  could add at the AudioSession level eventually.  0.5 Track G.
* **Two-pass STT cascade** (base.en for VAD/wake matching,
  medium.en for committed phrases) — VoiceLLM ships this; JROS
  config has `--stt-mode two_pass` flag but its behavior under
  the new AudioSession needs audit.  0.5.

## The unified pipeline target (post-refactor)

```
microphone (one owner, AudioSessionNode)
       │
       ▼
  WebRTC/energy VAD              ← acoustic gate (0.5 — Track G/H)
       │
       ▼
  Whisper STT                    ← STT model
       │
       ▼
  non-speech marker filter       ← already exists at non_speech.py
       │
       ▼
  duplicate suppression          ← NEW (0.4.x — small, deterministic)
       │
       ▼
  self-speech filter (default ON)  ← exists, default flip needed
       │
       ▼
  minimum length / syllable gate ← NEW (0.4.x — small, deterministic)
       │
       ▼
  publish /sense/transcript      ← bus boundary
       │
       ▼
  one gate consumer              ← Codex's classify_voice_input(),
  (input_gate.py)                  used by BOTH voice_loop AND
       │                            voice_session
       ▼
  if accepted:
    hand off to normal turn-running function
    (shared by typed input + voice)
  else:
    log "[gate: ignored: <reason>]" to active interface
    drop on the floor
```

**The key architectural commitment:** there is ONE gate
implementation, ONE filter chain, ONE turn-running function.
Interfaces (TUI, standalone, messaging gateway, future things)
are PRESENTATION ONLY.  They observe state, render output,
forward input.  They do not have their own filter logic.
