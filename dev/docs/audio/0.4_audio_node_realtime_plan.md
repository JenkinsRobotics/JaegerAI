# 0.4.0 realtime audio/node hardening plan

**Status:** proposal for implementing-agent review
**Date:** 2026-06-07
**Goal:** make the 0.4.0 node architecture line up with the long-term
robot shape without making the voice path slower than the VoiceLLM
reference.

## Verdict

The 0.4.0 node direction is right, but the audio ownership boundary is
not ideal yet.  JROS currently has a working TTS node and an STT node,
but it does not yet have one central audio/session node that owns mic,
speaker reference, AEC, VAD, STT, cleanup, follow-up state, and
barge-in as a single realtime subsystem.

The fix should not be "split every audio stage into a separate process."
That would add avoidable queueing, serialization, and scheduling latency
to the most latency-sensitive part of the robot.  The right shape is a
single realtime audio session node in monolithic mode, with internal
components for mic/AEC/VAD/STT/TTS-reference and typed bus events at
the semantic boundaries.

## Current state

### Operational today

- `TTSNode`
  - Subscribes `/act/speech`.
  - Subscribes `/act/speech_stop`.
  - Runs Kokoro through the node.
  - Publishes `/sense/spoken` ack.
  - Brain/TUI speech output can route through the bus.

- `STTNode`
  - Wraps the existing Whisper STT adapter.
  - Starts/stops the mic through that adapter.
  - Polls finalized phrases.
  - Publishes `/sense/transcript`.
  - Standalone `voice_loop.py` already uses this bus path.

- `VoiceController` in the TUI
  - Still directly creates and owns `WhisperSTTTwoPass`.
  - Still owns AEC/reference buffer setup.
  - Still polls STT directly for user phrases.
  - Uses the TTS node for speech output, but not the STT node for
    speech input.

### Limit today

The realtime voice path is split by caller:

- standalone voice loop: mostly node-shaped for STT/TTS
- TUI voice: direct STT ownership plus bus-routed TTS

That works, but it means 0.4.0 has a node foundation rather than a
fully coherent audio node architecture.

## Design principle

Do not move raw audio across the bus unless a real process/device
boundary requires it.

Raw mic frames at 16 kHz or 24 kHz are not large by robotics standards,
but they are too frequent for casual topic fan-out in the hot path:

- 20 ms audio frames = 50 messages/sec per stream
- AEC needs tight alignment between near-end mic and far-end speaker
  reference
- VAD/STT segmentation benefits from local buffers and direct callbacks
- Python queues plus serialization are fine for semantic events, not
  ideal for inner-loop audio timing

So the efficient boundary is:

- inside audio session: raw frames, AEC, VAD, STT buffers, callbacks
- bus boundary: finalized transcripts, speech commands, stop events,
  user-speaking events, health, optional low-rate diagnostics

## Target node shape

### `AudioSessionNode`

Proposed path:

`jaeger_os/nodes/audio_session/node.py`

This node should own the realtime voice session:

- mic capture backend selection
  - `avaudio`
  - `sounddevice` fallback
- speaker reference buffer for AEC
- AEC enable/fallback decision
- VAD / phrase segmentation
- STT engine
  - two-pass default
  - continuous optional
- non-speech marker filter
- self-speech filter
- stale phrase handling
- follow-up window state
- barge-in speech-detected callback
- pause/resume mic state
- transcript publishing

It should publish:

- `/sense/transcript`
  - finalized, cleaned user utterance
- `/sense/user_speech_start`
  - low-latency event for interruption/barge-in
- `/sense/audio_state`
  - optional health/state snapshot
- `/health/audio_session`
  - later Track D style health topic, if health topics land

It should subscribe:

- `/act/speech`
  - not to synthesize speech, but to know that agent speech is about
    to start if the TTS node cannot provide a better state event
- `/sense/spoken`
  - open follow-up window after successful speech
- `/act/speech_stop`
  - clear local speaking/interruption state
- `/control/mic_pause`
  - explicit external pause/resume
- `/control/stt_followup_open`
  - explicit follow-up window command

Important: TTS synthesis/playback can remain in `TTSNode`.  The audio
session node only needs the far-end reference stream or reference buffer
for AEC.  In monolithic mode, share this reference in process.  In
multiprocess mode, add a lightweight reference-feed topic only if it is
actually needed.

## Node boundary recommendation

### Keep together

These should live together in one realtime audio session node:

- mic input
- AEC
- VAD
- STT buffering
- STT finalization
- non-speech filtering
- self-speech filtering
- follow-up state
- barge-in detection

Reason: splitting these into separate nodes creates latency and makes
AEC alignment harder.

### Keep separate

These should remain separate nodes:

- TTS synthesis/playback
- vision/camera
- motor
- lights
- brain/agent loop
- background/deep-think work

Reason: they are naturally independent work queues and can tolerate bus
boundaries.

### Optional later split

Only split raw `audio_in` and `audio_out` into independent nodes when a
real hardware/process boundary needs it:

- remote mic board
- Jetson-hosted audio preprocessing
- separate speaker device controller
- cross-machine robot body

Until then, keep raw frames inside the audio session node.

## Execution order

### Step 1 - Rename the architecture target

Clarify docs and code comments that the 0.4.0 target is not a generic
`audio_io` node that blindly forwards frames.  The target is an
`AudioSessionNode` that owns the realtime voice input session.

Suggested doc update:

- `audio_io`: low-level future raw frame device node
- `audio_session`: current 0.4.0 voice session node

This avoids designing the hot path around a lowest-level abstraction
too early.

### Step 2 - Extract a shared audio session adapter

Before changing bus wiring, extract the common logic currently spread
across `VoiceController` and `voice_loop.py`:

- build AEC/reference buffer
- build Whisper STT in selected mode
- configure audio backend
- pause/resume mic
- open follow-up
- drain pending
- install speech-detected callback
- apply non-speech filter
- apply self-speech filter

Suggested path:

`jaeger_os/core/audio/session.py`

This should be a plain Python class first, not a node.  The goal is to
remove duplicate ownership logic before introducing a second lifecycle
layer.

Acceptance:

- TUI and standalone voice can both construct the same session object.
- No behavior change yet.
- Existing tests still pass.

### Step 3 - Implement `AudioSessionNode`

Wrap the shared session object in a node:

- `setup()` starts mic/STT.
- `tick()` polls finalized phrases and publishes `/sense/transcript`.
- speech-start callback publishes `/sense/user_speech_start`.
- `teardown()` stops mic/STT and clears callbacks.

This is the STT node successor for voice mode.  The existing `STTNode`
can remain as a narrow adapter node for tests and simple pipelines, but
the TUI should use `AudioSessionNode` because it needs AEC/follow-up/
barge-in semantics.

Acceptance:

- Node can be started in-process.
- Publishes transcripts.
- Publishes user-speech-start quickly during barge-in.
- Can pause/resume mic via method first, then topic in Step 5.

### Step 4 - Move TUI input to `/sense/transcript`

Change `VoiceController` so it no longer owns `WhisperSTTTwoPass`
directly.  It should own or reference an `AudioSessionNode`, subscribe
to `/sense/transcript`, and expose `poll()` from a small local queue.

The TUI can still be the UI orchestrator:

- display voice status
- decide when to submit turns
- coalesce busy voice turns
- speak replies through `/act/speech`

But it should stop owning mic/STT internals.

Acceptance:

- TUI voice and standalone voice both consume the same transcript topic.
- No second mic owner exists in a normal TUI session.
- TUI stop tears down the audio node cleanly.

### Step 5 - Replace direct STT control calls with topics

After Step 4 works, replace direct method calls with control topics:

- `/control/mic_pause`
- `/control/stt_followup_open`
- `/control/audio_drain_pending`
- possibly `/control/audio_mode`

Keep these as control topics, not `/act/*`, because they coordinate the
audio session rather than express robot actions.

Acceptance:

- TUI and standalone voice no longer need a direct STT object reference.
- Tests can assert pause/follow-up/drain via bus messages.

### Step 6 - Add cheap pre-LLM voice gate

Keep the existing LLM gate, but do not make it the first meaningful
filter.

Add a pre-LLM gate inside the audio/session or voice orchestrator path:

1. Empty/junk marker gate
2. Minimum content gate
3. Repeated transcript gate
4. Stale phrase gate
5. Busy-turn coalescing
6. Optional small intent classifier
7. Existing LLM `<ignore>/<reply>` gate

This is the "agentic lite" layer.  It should be cheap and deterministic
by default.  If a tiny model/classifier is added later, it should be
optional and measured against the rule gate.

Acceptance:

- Clicks, blanks, filler artifacts, and repeated stale phrases do not
  consume full LLM turns.
- If five voice fragments arrive while the agent is busy, they become
  one next-turn batch or are dropped as stale, not five sequential LLM
  turns.
- The full LLM gate remains the semantic backstop for "was that
  addressed to me?"

### Step 7 - Add turn routing classes

Voice should not route every accepted phrase into the same slow path.

Add an early route classification:

- `immediate`
  - short factual answer
  - local status
  - simple memory lookup
  - simple web/search request with freshness requirement
- `clarify`
  - ask one follow-up question quickly
- `background`
  - coding
  - research
  - code review
  - long benchmark
  - large tool workflow
- `hardware`
  - robot body command, safety-gated

This does not require a separate model first.  Start with rules and
tool-intent heuristics, then promote to a small router only if needed.

Acceptance:

- "Search for X" acknowledges quickly and begins fresh lookup.
- "Do a code review" can become a background task after any required
  clarifying question.
- Voice loop remains responsive while background work continues.

### Step 8 - Context staging and compaction

Do not load the full brain context for every voice fragment.

Add context levels:

- `voice_minimal`
  - identity
  - active conversation summary
  - realtime commands
  - small toolset
- `default`
  - current normal agent context
- `deep_task`
  - coding/research skill docs
  - long memory
  - larger toolset
  - background board/deep-think integration

Acceptance:

- Simple voice turns use a smaller prompt/toolset.
- Deep tasks can still expand context after routing.
- Completed background work summarizes back into memory instead of
  bloating the active voice prompt.

## Test plan

### Unit tests

- `AudioSession` builds correct STT mode and audio backend.
- AEC unavailable falls back to mic-pause.
- Non-speech markers are dropped before transcript publish.
- Self-speech filter drops near-duplicate agent replies.
- Stale phrase policy drops old phrases.
- Busy coalescing merges multiple pending voice phrases.
- Control topics pause/resume/open follow-up/drain pending.

### Node tests

- `AudioSessionNode` publishes `/sense/transcript`.
- `AudioSessionNode` publishes `/sense/user_speech_start`.
- Teardown calls adapter stop exactly once.
- No transcript publish after stop.
- Bounded transcript queue drops old phrases, not new realtime input.

### Integration tests

- TUI voice starts one mic owner only.
- TUI receives transcript via bus.
- Standalone voice and TUI use same audio session implementation.
- Barge-in publishes `/act/speech_stop`.
- TTS node ignores stale correlated stop.
- Mic-pause path does not queue agent self-speech.

### Operator smoke gates

- `./launch --tts-test`
- TUI voice: ask a simple question, get immediate response.
- TUI voice while agent is speaking: barge-in stops speech.
- TUI voice while agent is thinking: multiple user fragments coalesce.
- Background task: voice asks for code/research work, agent acknowledges
  quickly and creates/continues background work.

## Performance budget

Targets for 0.4.0:

- transcript publish after STT finalization: under 20 ms local overhead
- barge-in speech-start to `/act/speech_stop`: under 50 ms local overhead
- no raw audio topic fan-out in monolithic default
- no full LLM turn for known junk transcripts
- no unbounded phrase queue
- no more than one active mic owner per process

## Non-goals for 0.4.0

- Full cross-machine raw audio transport
- Separate mic process by default
- Streaming every audio frame over ZMQ in monolithic mode
- Replacing Whisper STT
- Replacing Kokoro TTS
- Building a full router model before the rule-based route works

## Recommended implementation order summary

1. Document target as `AudioSessionNode`, not generic raw `audio_io`.
2. Extract shared `AudioSession` from TUI/voice_loop duplication.
3. Wrap it in `AudioSessionNode`.
4. Move TUI input to `/sense/transcript`.
5. Convert direct STT control calls to control topics.
6. Add cheap pre-LLM gate and stale/coalesced queue policy.
7. Add turn routing classes: immediate, clarify, background, hardware.
8. Add context staging: voice_minimal, default, deep_task.

This preserves the node architecture while keeping the hot audio loop
inside one efficient realtime subsystem.
