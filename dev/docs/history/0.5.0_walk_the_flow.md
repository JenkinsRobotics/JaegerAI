# 0.5.0 — walk the flow (operator verification)

**Status:** ship gate — 0.5.0 is `0.5.0` only after this passes
**Memory anchor:** `feedback-walk-flow-before-shipping`

> *Never ship installer/wizard/UX work on inspection alone; actually
> walk the flow.*

This doc is the checklist the operator runs before the 0.5.0
release tag.  Each step has a pass criterion.  If any step fails,
0.5.0 isn't shippable — file a bug, fix it, re-run the whole
checklist.

## Prerequisites

- Branch `origin/0.5.0` checked out at tip (`4544ce0` or later)
- `./install.sh` has run successfully
- Working microphone + speakers
- macOS — at least one supported audio backend
- Optional: OBS installed (for the streaming verification step)

## Step 1 — Boot the avatar pipeline

```bash
./launch --instance jros-dev
```

**Watch for in the boot log:**
```
[jaeger] warmed avatar (animation node + bridge) in 0.Xs
```

**Pass:** the warm line appears and `./launch --tts-boot-test`
+ `./launch --node-test` both still pass.

**Fail mode to watch for:** missing warm line → `config.avatar.enabled`
is false, or `--no-avatar` was passed.

## Step 2 — Connect the Swift renderer

In a second terminal:

```bash
cd apps/JROS-Avatar
swift run JROSAvatar
```

The Mac app window opens with:
- A text field showing `ws://127.0.0.1:8765/frames`
- A "Connect" button
- The "Waiting for frames…" placeholder

Click **Connect**.

**Pass:** status indicator goes green, "Disconnected" → "Connected",
frame count still 0.

**Fail mode to watch for:** connect fails → AnimationNode didn't
spin up.  Check the JROS boot log.

## Step 3 — Smoke-test the face manually

In a third terminal (or the JROS TUI's `>` prompt if you're in
chat mode), trigger a state change:

```bash
.venv/bin/python -c "
from jaeger_os.nodes import runtime
runtime.ensure_animation_node()
from jaeger_os.agent.tools.avatar import set_avatar_state
from jaeger_os.agent.tools import _common
from jaeger_os.core.instance.instance import (
    InstanceLayout, default_instance_name, resolve_instance_dir,
)
layout = InstanceLayout(root=resolve_instance_dir(default_instance_name()))
_common.bind(layout)
print(set_avatar_state(emotion='happy'))
"
```

**Pass:** the Swift app shows Lilith's face — soft luminous oval,
big smile, cheek flush, squinted eyes.  Breathing pulse subtly
moves the face up + down.  Blinks every ~4 seconds.

**Fail mode to watch for:** the placeholder stays → frames aren't
arriving.  Check the WebSocket connection status + the
AnimationNode's logs.

## Step 4 — Cycle through emotions

Same shell session — cycle the emotions:

```bash
.venv/bin/python -c "
from jaeger_os.agent.tools.avatar import set_avatar_state
from jaeger_os.agent.tools import _common
from jaeger_os.core.instance.instance import (
    InstanceLayout, default_instance_name, resolve_instance_dir,
)
layout = InstanceLayout(root=resolve_instance_dir(default_instance_name()))
_common.bind(layout)
import time
for emotion in ('neutral', 'happy', 'sad', 'focused', 'thinking',
                'speaking', 'listening'):
    print(f'→ {emotion}')
    set_avatar_state(emotion=emotion)
    time.sleep(1.5)
print('done')
"
```

**Pass:** Lilith's face cycles through all 7 emotions over ~10
seconds.  Mouth shape changes per emotion.  Eye shape changes
(squinted on happy, narrowed on focused, wide on listening).
Brow lifts on thinking.  Cheek flush appears on happy.

## Step 5 — REAL-TIME CONVERSATION (the ship gate)

This is the actual 0.5.0 acceptance test.

```bash
./launch --stream --instance jros-dev
```

**Watch the boot log:**
```
[jaeger] streaming mode — open the Swift renderer at:
          apps/JROS-Avatar (swift run JROSAvatar)
        connect URL:
          ws://127.0.0.1:8765/frames
```

In the Swift app (still running from Step 2 — reconnect if needed),
ensure connection is green.

Speak to JROS naturally: "Hello, Lilith. Tell me what you see."

**Pass criteria (ALL of these must hold):**

1. ⬜ Whisper picks up your speech (transcript appears in the JROS
   terminal)
2. ⬜ The brain generates a response in reasonable time
3. ⬜ At the moment TTS starts speaking:
   - Lilith's face automatically switches to "speaking" emotion
   - Her mouth opens + closes in time with the speech
   - The lip-sync feels natural enough that you can't tell it's
     a sin-wave proxy at a glance
4. ⬜ When TTS finishes:
   - Lilith returns to "neutral" emotion
   - Mouth closes back to a subtle smile
   - Breathing + blinking continue
5. ⬜ You can have multiple back-and-forth turns and the cycle
   repeats cleanly each time
6. ⬜ No CPU spike beyond ~30% during the conversation (check
   Activity Monitor — animation rendering should be cheap)
7. ⬜ No frame drops visible to the eye (Swift app's frame count
   advances steadily)

**Fail modes worth diagnosing:**

| Symptom | Likely cause |
|---|---|
| Speech happens but face doesn't change | AvatarAutoStateDriver didn't subscribe — check `ensure_animation_node()` ran |
| Face changes but mouth doesn't move with speech | `/sense/tts_chunk` events not arriving at AnimationNode — check TTSNode's _tts_amplitude_pulse thread |
| Face returns to neutral too late | `/sense/spoken` ack arrives late — check TTSNode's _publish_ack timing |
| Lilith looks frozen (no breathing/blinks) | MathScript stopped rendering — adapter probably crashed silently |
| Crackling audio | TTS amplitude thread spinning too hot — check CPU |

## Step 6 — OBS capture (the streaming use case)

This is the "make it ready for YouTube" test.

1. Open OBS
2. Add a Browser Source (or Window Capture for the Swift app)
3. Point at the Swift app's window
4. Run a 60-second conversation (Step 5 again, but extended)
5. Stop OBS recording + watch the playback

**Pass:** the recording shows Lilith speaking + reacting cleanly.
Audio + video stay in sync.  No visible stutters.

## Step 7 — Resource sanity

After 5 minutes of conversation, check:

- `ps -O rss` on the JROS process: under 5 GB resident (12B
  Gemma + Kokoro + Whisper + Lilith).
- Activity Monitor CPU: idle should be ~5%, conversation peaks
  ~30-50%, sustained.
- No memory growth over time (run the same 60s conversation
  3 times; resident shouldn't climb).
