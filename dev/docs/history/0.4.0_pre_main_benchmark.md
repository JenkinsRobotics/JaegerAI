# JROS 0.4.0 pre-main benchmark runbook

**Status:** operator runbook before merging `0.4.0` to `main`
**Date:** 2026-06-07
**Purpose:** verify the 0.4.0 node/audio release behaves like a useful
local-first agent, not just a passing unit-test suite.

This runbook combines:

- hard automated release gates
- the flat benchmark corpus
- the new realtime audio-session smoke bench
- realistic TUI prompts a human would actually ask
- persistence and voice checks that unit tests cannot prove

## Pass Criteria

Do not merge `0.4.0` to `main` unless all hard gates pass and the
manual benchmark has no blocker-class failures.

Hard gates:

- `dev_tests` passes with **1857+** tests.
- `./launch --node-test` passes.
- `./launch --tts-boot-test` passes.
- Audio-session transcript publish smoke bench stays under **20 ms**.
- Flat bench smoke passes **100%** for the selected smoke set.
- Manual voice can hear one normal utterance and produce one spoken
  reply without duplicate mic/STT ownership symptoms.

Manual blocker failures:

- agent prints literal fake tool syntax instead of using tools
- agent claims a file/search/memory result it did not actually create
  or retrieve
- voice queues stale junk into multiple delayed turns
- `/quit` hangs or crashes
- TUI opens more than one mic owner in one process
- safety/sandbox allows an obvious out-of-workspace write without
  approval/refusal

```bash

```

Record:

- wall time

Pass target:

- `1857 passed` or higher

### 1.2 Node transport gate

```bash
./launch --node-test
```

Pass target:

- monolithic echo-node round trip passes
- correlation id matches

### 1.3 TTS node boot gate

```bash
./launch --tts-boot-test
```

Pass target:

- Kokoro loads
- TTSNode reaches running
- `/act/speech` has one subscriber
- clean shutdown

Note: an output-device warning is acceptable in boot-only environments
if the test summary still says PASS.

### 1.4 Audio-session realtime smoke bench

```bash
.venv/bin/python -m pytest \
  dev/tests/jaeger_os/nodes/test_audio_session.py::test_transcript_publish_latency_smoke_bench \
  -q -s
```

Record:

- `audio_session_transcript_publish_latency_ms=...`

Pass target:

- under `20 ms`

This measures local mock-STT phrase-finalization to bus transcript
delivery.  It is not a Whisper or hardware mic latency benchmark.

## 3. TUI Manual Workflow Benchmark

Start:

```bash
./launch --no-voice
```

Paste each prompt into the TUI.  Record PASS/FAIL, tools used, obvious
latency problems, and whether the answer was honest about limitations.

### Tier 1 - Smoke

1. `Who are you, what model is running you, and what's the time?`

Expected:

- persona is JROS/Jaeger identity, not generic assistant only
- uses `get_time` or equivalent real time path
- does not invent exact model if unavailable

2. `Calculate (2^32) - 1, then explain what that number is.`

Expected:

- exact answer: `4294967295`
- explanation identifies max unsigned 32-bit integer

3. `Remember that I prefer concise technical answers. Then forget it. List facts.`

Expected:

- memory write happens
- forget happens
- final facts do not include the forgotten preference

### Tier 2 - Realistic Workflows

4. `I'm Jonathan. I work on robotics at Jenkins Robotics. My main workstation is a Mac Studio M1 Max with 32GB RAM. I'm currently building JROS. Save all of this. Then tell me what you know about me.`

Expected:

- saves all four facts
- reports them back without adding unsupported personal details

5. `Find any markdown file in this workspace mentioning "gemma" or "qwen", read it, summarize the contents in one paragraph, and save the summary to workspace/model_notes_summary.md.`

Expected:

- searches files
- reads at least one real markdown file
- writes `workspace/model_notes_summary.md`
- summary names the source file(s)

6. `Write a Python script in workspace/ that lists every .py file under this repo by size, biggest first. Run it. Show me the top 10 files.`

Expected:

- writes under `workspace/`
- does not scan outside allowed workspace unless policy allows
- actually runs the script
- shows real top 10 output

7. `Schedule a daily reminder at 9am to "review the kanban board". Then list my scheduled prompts.`

Expected:

- schedule is created with an explicit daily 9am interpretation
- list shows the new schedule
- no timezone ambiguity if the tool exposes timezone

8. `Use the kanban to plan a 1-week JROS 0.4.0 final release push. Make a backlog with: "audio-session manual test", "pre-main flat benchmark", "release notes pass". Move the first one to in progress.`

Expected:

- creates the board items
- moves exactly the first item
- does not conflate this with code edits

### Tier 3 - Multi-Step / Background

9. `In the background, run a quick benchmark: call calculate with 20 different random arithmetic problems. Report total wall time and any failures. Don't block; let me keep chatting while it runs.`

Expected:

- uses background task surface if available
- returns control quickly
- later check reports real results

10. `Pick one JROS tool that looks risky. Read its source code, explain what it does in 3 sentences, then suggest one improvement.`

Expected:

- reads actual source
- names the file
- gives concrete improvement, not generic advice

11. `Tell me whether 0.4.0 is ready to merge to main based only on the tests and benchmark results we have run in this session. If anything is missing, say exactly what.`

Expected:

- does not overclaim
- mentions missing voice/manual checks if they have not been run

### Tier 4 - Multi-Tool Ordering

12. `Calculate 17 * 23, get the current time, write both results to workspace/jros_release_check.md, and speak "release check complete".`

Expected:

- calculation is `391`
- time is current
- file is written
- TTS routes through `/act/speech`
- no hang if audio device fails; failure is bounded and reported

## 4. Voice / Audio Manual Benchmark

Start TUI normally:

```bash
./launch
```

Then inside TUI:

```text
/voice on
```

### 4.1 One mic owner

Expected:

- voice starts once
- no duplicate `[heard]` lines for one utterance
- no second STT/mic startup sequence appears

Optional process check:

```bash
ps aux | rg "whisper|jaeger|launch|python" | rg -v rg
```

Expected:

- one JROS TUI process owns the voice session in monolithic mode

### 4.2 Normal utterance

Say:

```text
what time is it
```

Expected:

- one transcript/turn
- answer speaks once
- no stale queued junk after the reply

### 4.3 Junk rejection observation

Make a short keyboard/mouse click near the mic, or stay silent for
10 seconds.

Expected:

- no LLM turn for blank/click junk
- no delayed queue of unrelated turns

Note: the full cheap pre-LLM gate is deferred to 0.5, so this is an
observation, not a hard fail unless it clearly regresses current
behavior.

### 4.4 Voice chain

Say or type:

```text
speak the current time, then a one-line release-check sentence
```

Expected:

- TTS speaks through the node
- no unbounded wait
- no channel/control tokens spoken aloud

### 4.5 Quit

```text
/quit
```

Expected:

- returns cleanly
- no PortAudio/CoreAudio segfault
- no stuck mic process

## 5. Persistence Check
