# Code review request — JROS 0.4.0

You're reviewing JROS 0.4.0, the node-architecture release of an
operator-built local-first agentic AI framework.  This is an
independent code review — the implementing agent is asking you to
catch what they missed.  Bias toward honesty over politeness.  If
the design is wrong, say so.  If a layer is overengineered, say so.
If a test is checking the wrong thing, say so.

## Context

JROS positions itself as **ROS + Agentic AI + Mac-first local
hardware**.  Brain (Mac, local Gemma + tools + memory + skills) is
one process; peripheral subsystems (TTS, STT, vision, motors,
lights) are bus-addressable nodes behind clean adapter Protocols.

0.4.0 is the architectural release that introduces the node shape.
Before this, voice mode was direct in-process calls; now it routes
through a typed Bus with msgspec.Struct schemas and a request/ack
correlation_id pattern.

## What to review

Branch: `0.4.0`
Tag: `0.4.0`
Compare baseline: `0.3.0` (the previous release, in-process TUI
without nodes)

Specifically:

1. **Architecture coherence** (Track A — transport + nodes)
   - `jaeger_os/topics.py` (msgspec.Struct topic SSOT)
   - `jaeger_os/transport/codec.py` (JSON+MessagePack picker)
   - `jaeger_os/transport/bus.py` + `inproc_bus.py` + `zmq_bus.py`
   - `jaeger_os/transport/broker.py` (XPUB↔XSUB proxy)
   - `jaeger_os/nodes/base.py` (Node base class)
   - Is the abstraction earning its keep, or is it overengineered
     for a Mac-only / one-developer use case?
   - Does the Bus API give the brain what it needs without
     leaking transport details into the agent layer?
   - msgspec was chosen over Pydantic for the hot path — was that
     the right call?

2. **Brain integration** (Track B — node implementations + brain
   wire-up)
   - `jaeger_os/nodes/{tts,stt,vision,motor,light}/`
   - `jaeger_os/nodes/runtime.py` (brain singleton)
   - `jaeger_os/core/tools/speak.py` (text_to_speech tool now
     publishes /act/speech)
   - `jaeger_os/plugins/voice_loop.py` (full migration: phrases on
     /sense/transcript, TTS via bus.request, barge-in via
     /act/speech_stop)
   - `jaeger_os/interfaces/tui/voice_session.py` (TUI voice path
     w/ LLM gate parsing)
   - Does the "tool does the networking, node does the execution"
     contract actually hold, or are there leaks where the brain's
     tool layer reaches around the bus?
   - Are the per-subsystem Protocol interfaces (`Synthesizer`,
     `STTAdapter`, `CameraAdapter`, `MotorAdapter`, `LightAdapter`)
     the right shape for swappable hardware backends?

3. **Voice-mode defaults change**
   - Commit `517d751` flips wake_word OFF + llm_gate ON + barge_in
     OFF by default to match the proven VoiceLLM reference pattern.
   - Is this the right tradeoff for fresh installs?  What surprises
     could it cause an operator?

4. **Test surface**
   - 1824 tests total; +149 new vs 0.3.0.
   - `dev/tests/jaeger_os/transport/` (codec / InProcBus / ZMQBus /
     broker)
   - `dev/tests/jaeger_os/nodes/` (base / runtime / tts / stt /
     vision / motor / light)
   - Are critical paths actually tested?  Are there gaps where the
     test relies on integration verification (e.g.
     `./launch --tts-test`) instead of unit coverage?
   - Look especially at `dev/tests/jaeger_os/nodes/test_runtime.py`
     — implementing agent flagged a pytest fixture deadlock with
     mock-based `ensure_tts_node` testing and trimmed the test to
     5 simple cases.  Is the trimming legitimate, or hiding a real
     bug?

5. **Known unfinished** (don't review for completeness — review
   the DESIGN choices behind these being unfinished)
   - `./launch --mode multiprocess` end-to-end is gated by the
     broker work but launch.py spawning is a 0.4.x followup.
     Is the design sound?
   - `audio_io` node split (B.4) — STT currently owns the mic
     via WhisperSTTContinuous.  Is the deferral defensible?
   - Voice control topics (`/control/mic_pause`,
     `/control/stt_followup_open`) — voice loop still calls these
     methods directly on the STT engine reference.  Worth porting
     to topics now or wait?

## Specific things implementing agent wants you to challenge

1. **msgspec everywhere in transport schemas, Pydantic for
   config.**  Is the boundary defensible, or should it be all one
   or all the other?

2. **Voice loop migration kept `tts.stop()` direct-call at
   teardown** even though the in-flight TTS now goes through the
   bus.  Is that a leak that should be plumbed through
   `/act/speech_stop` too?

3. **TTS node's `_on_speech_stop` calls `synthesizer.stop()`
   unconditionally** — no correlation_id matching.  Is that right
   for barge-in semantics, or could it interrupt a SECOND
   in-flight speech that wasn't the one the operator meant to
   interrupt?

4. **VisionNode publishes raw camera frames on
   `/sense/vision`** — no inference, no scene description.  Is the
   operator's "vision = stream, not analysis" call right, or does
   the topic name mislead future consumers who'd expect analysis?

5. **Track C motor + light Protocols use velocity twists + RGB
   patterns.**  Is the universal interface too generic to be
   useful for real boards, or about right for the planned
   instance-level adapter pattern?
